# session 21 of building The Joy in the open

## where session 20 left things

The last session was the one where the loop closed. It began on a flaw baked into lazy minting — that routing concepts by similarity is bound, over enough facts, to breed semantic duplicates, two type-names for one idea, and that this can only ever be cleaned up *behind* the write path, never prevented on it. So two bricks went in against that. First, two clocks on every fact: `created_at` for when we were told and a new nullable `happened_at` for when the thing actually happened, with the null standing in as `created_at` at read time — one particular, time, promoted out of the thin raw payload because nearly every question a diary is asked is a question about time. Then an offline merge pass that finds near-twin types by vector nearness, asks the model one pair at a time whether they are genuinely the same kind, and collapses the confirmed families onto a survivor, leaving the losers as redirects rather than deletions.

With the write path whole, the session turned to the half it had been arguing for all along — the read path — and got the fast half of it standing: a lexical librarian reaching into the raw facts through Postgres full-text search under both English and French analysers, and the first reply the kernel ever composed off the diary in its own voice rather than a canned line. Then the seams that made it real: live ingestion wired in at last, so every symbiot message files itself into the diary exactly once through a background sweep, decoupled from the reply so the human never waits on bookkeeping; the whole path driven by hand through the shell in a browser against a live kernel, security seam and authed path both watched from the database; a move onto the client Ollama actually publishes; and finally a real engine behind the loop — GLM-5.2 on Scaleway's GPU-backed inference with a fallback ladder home through Mistral to the local Qwen, output ceilings and sampling both pinned rather than left on provider defaults. The conversational loop now runs end to end, live, from the shell the symbiot will actually use.

## what this session is about: whether the shell needs xterm at all

This session opens not on the kernel but on the shell — and not on a feature to add but on a dependency to question. The shell renders its terminal with [xterm.js](https://xtermjs.org/), the same emulator that backs VS Code's integrated terminal: a full VT/ANSI terminal, canvas renderer, fit addon, the works. It is a genuine powerhouse. The doubt is whether that is a power The Joy's shell has any use for.

The doubt has a concrete source. I was walking outside, using the shell on my phone, and watched it misbehave — several display bugs on the small touch surface, the kind of thing xterm handles cleanly on a desktop and evidently less so on a phone held one-handed on the move. And the phone is not an edge case here; it is very close to the primary case, because talking to the diary is a thing you do in the ordinary flow of a day, standing in a queue or walking a street, not only sat at a desk.

So it is worth asking what xterm is actually earning its place doing. The shell already keeps its own line buffer, echoes its own keystrokes, prints its own prompts and ghost suggestions by hand — it drives the terminal through raw ANSI escapes rather than leaning on much of what xterm offers above the render surface. What xterm is really providing is a faithful *terminal*: cursor addressing, scrollback, a VT emulation complete enough to run programs that expect a real TTY. But The Joy's shell runs no such programs. It reads a line, sends it to the kernel, prints the reply. For a good while yet — likely a long while — what it needs from the terminal is not fidelity but a *feel*: a monospaced prompt, a cursor, text that scrolls, the unmistakable sense of talking to something at a command line. That feel is a much smaller thing than a real terminal emulator, and it may well be a thing better served by plain DOM I control end to end than by a heavyweight emulator I fight to behave on the surface that matters most.

That is the question this session starts by answering, honestly, before writing anything: is xterm worth keeping, or is the terminal feel cheaper and *better* built by hand — especially on the phone. Not a foregone conclusion in either direction. The evaluation comes first; the brick, if there is one, comes after it.

## how the evaluation runs, before any verdict

The decision earns four steps, in order, and the point of the order is that the verdict lands against evidence rather than instinct. First, take an honest inventory of what xterm actually earns in the shell as it stands — every terminal surface the code truly touches, `write` and its ANSI sequences, `onData`, the fit addon, the canvas renderer, the column and row counts — so we are weighing the dependency against what it *does for us*, not against everything it is capable of. Second, reproduce and characterise the phone bugs that started this, because the fix hinges on their cause: display bugs that turn out to be a misconfigured renderer or a fit-sizing quarrel with the software keyboard are an argument to *tune* xterm, while bugs structural to running a full VT emulator on a one-handed touch surface are the strongest argument to drop it. Third, sketch the hand-rolled replacement's real shape — a plain DOM and CSS terminal *feel*, a scrolling log, a monospaced input line with a cursor, the ghost suggestions, the line-read modality the auth flow needs — and name plainly what we would rebuild, what we already own, and what we would lose. Fourth and last, weigh it and recommend: feel against fidelity, phone first, with the reasoning laid out to be argued with rather than a conclusion handed down.

Only after that verdict is confirmed does any code move. What follows in this log picks up at step one.

## the inventory: what xterm was actually earning

The first honest look was at how much of xterm the shell truly leans on, and the answer was: almost none of what makes xterm a *terminal*. Three files touch its API, and they are wildly unequal. Two others — the auth flows and the notify flow — never touch it at all, because they were already written against a two-method `readLine`/`print` seam rather than the terminal itself, so the abstraction that this session needed already half-existed. Of the three that do touch it, one is trivial (a screen-clear and a line-write), one is shallow but broad (the launch: constructing the terminal, the fit addon that sizes the grid to its box, the canvas renderer chosen for paint speed, one read of the column count to pick the banner, and the line editor — which already keeps *its own* buffer and echoes by hand, driving xterm purely as a dumb render surface through ANSI escapes), and exactly one leans on xterm as a real emulator: the capture loop, which repaints a queued line's marker in place by reading the cursor's absolute row out of xterm's screen buffer and moving the cursor around with save/restore and erase escapes.

That last one is the whole of the deep coupling, and naming it precisely is what made the rest of the decision easy. The shell drives no program that expects a terminal; it emits only a handful of its own colour codes and reads back only the cursor row for that one repaint. Everything xterm exists to provide above a monospaced scrolling surface — VT emulation, scrollback semantics, wide-character handling, a real cursor model — is either unused or used in exactly one place that a plain DOM node reference would serve *more* simply.

## the phone bugs, and why they are xterm's shape, not our settings

Then the bugs that started the session, characterised against what actually broke on the phone: text that can't be selected to copy, a backspace that sometimes does nothing, touch-scrolling that fights back. All three trace to the same two choices *inside* xterm, not to anything misconfigured on our side. It paints the terminal to a `<canvas>` — which has no text nodes at all, so the browser has nothing to select and native copy-paste physically cannot reach it — and it captures typing through a hidden `<textarea>` that mobile soft keyboards read as empty, so the keyboard decides there is nothing to delete and swallows the backspace. The touch-scroll trouble is the same canvas viewport getting in the way of the native gesture.

This is the distinction the evaluation was built to draw, and it fell on the far side of it: these are not settings we forgot to flip. Switching the renderer back to DOM would recover some selectability, but the swallowed backspace lives in the input layer and would survive the switch untouched. The bugs are consequences of running a canvas-painted, hidden-textarea emulator on a one-handed touch surface — which is to say they are xterm's shape, and the only clean way past them is to stop using it.

## the verdict: drop it, and build the feel by hand

So the recommendation, made against the evidence rather than instinct: drop xterm. For a tool used mostly from a phone, that wants a terminal *feel* and no terminal *fidelity*, xterm is a net loss — it costs the precise interactions that broke (selection, backspace, scroll) in exchange for emulation the shell has never once used. The replacement is plain DOM and CSS, and it is small because the hard part was already ours: the line buffer, the echo, the prompt, the ghost suggestion are all hand-written already; xterm was only the surface they painted on and the pipe they read keystrokes from. A scrolling column of real text lines makes selection and touch-scroll work for free; a real editable input element makes the mobile keyboard — and its backspace — behave like every other web input, because now there is genuine text for it to see. The one piece of real emulation, the in-place marker repaint, gets *simpler*: instead of arithmetic over a screen buffer, the capture loop keeps a handle to the marker's own element and sets its text. What is given up — scrollback semantics, VT emulation, wide-character handling, canvas paint speed at scale — is exactly the fidelity a diary-at-a-prompt never draws on. The bottom-of-screen furniture the shell already had — the online/offline pill, the version tag, the three source links — was never xterm's to begin with, sitting in its own fixed layer over the terminal, so it stays exactly as it was.

## the build: the terminal by hand, and the three bugs gone with xterm

The replacement went in, and it is as small as the verdict predicted, because the hard parts were already ours. `term.ts` is now a plain-DOM terminal: a scrolling log of real text lines and a single `contenteditable` input line beneath it, with the one place that turns the shell's ANSI colour codes into styled spans kept intact so no caller had to change how it colours text. The shell still writes `\x1b[92m…\x1b[0m` and its friends; that vocabulary outlived the emulator that used to paint it. The line buffer, the echo, the prompt, the ghost suggestion, the modal line-read the auth flow leans on — all carried straight over, because none of them were ever xterm's to hold.

The three phone bugs that started the session are gone, and gone for free rather than patched around. The log is real selectable text, so selection and native copy work; the scroll is the browser's own, so the touch gesture stops fighting a canvas viewport; and the input is a genuine editable element the soft keyboard can see, so backspace behaves the way it does in every other web field. The one piece of real emulation — the capture loop's in-place marker repaint — got simpler exactly as promised: instead of reading a cursor row out of a screen buffer and moving it with save and restore escapes, `writeLine` now hands back the line's own DOM node, and the capture loop keeps that handle and repaints a marker by setting its text. `xterm` and its addons are out of `package.json`; the version moved to 0.0.2 on the way through.

## the banner, and a bug only the phone could show

Building the feel by hand meant owning a problem xterm used to hide, and the launch banner is where it surfaced. The banner is a block-glyph wordmark — figlet's "ANSI Shadow" — chosen because the thin line-art fonts turn to mush at small sizes in green-on-black while solid blocks stay legible. It ships in three cuts, widest first: a wide one-liner, a stacked two-word block a phone can hold, and a plain spaced-out wordmark for anything narrower than that. The shell has to pick the widest cut that actually fits the box it is drawn into.

The first version of that choice measured a candidate's width ahead of drawing it, and on the phone it chose wrong — the wide one-liner, which then rendered clipped at "THE JO" with the rest sheared clean off. The cause was timing, not arithmetic. The block and box-drawing characters resolve to a fallback font that lands late and wider than the primary, so a width measured before that font settled judged the wide cut as fitting, and it only overflowed once the real glyphs painted — and because banner rows are set never to wrap, an overflow clips rather than folds. The first signal I checked the fix against was no better placed to catch it: it asked whether a row had wrapped, which a non-wrapping row never does, so a clipped banner passed clean. The signal that actually catches it is overflow, not wrap — a rendered row wider than the box it sits in.

So the fix measures the thing that renders. `main.ts` now waits for `document.fonts.ready`, then paints the widest cut and asks the painted rows themselves whether they overflow — `scrollWidth` past `clientWidth` is the line being clipped — tearing the cut down and stepping to the next narrower one when it does, and keeping the plain wordmark as the floor there is nothing beneath. No separate width estimate is left to disagree with the paint, so whatever font a browser resolves those glyphs to, the shell settles on a cut that genuinely fits in that environment. Driven across widths from 900 down to 140 pixels it clips at none of them and steps cleanly wide → stacked → plain. The honest remaining check is the one surface that exposed the bug in the first place — the phone in hand, after deploy — because that is the font environment no desktop run reproduces.

## a gap in the fast reply: it remembers the diary but not the conversation

A second thread opened this session, away from the shell and back in the kernel, and it began as a plain question about what the quick reply actually carries. When the symbiot says something, the fast pass composes its answer from three things: the persona that sets its voice, the current message, and a lexical reach into the diary for the facts that bear on what was said. That is long-term memory working exactly as designed — the reply can lean on anything ever filed. But take one step past it and a hole opens where there should be none: the fast pass holds no short-term memory of the conversation it is sitting inside. It sees the diary and the single line in front of it, and nothing of the last few turns.

That is a real gap, not a theoretical one. A conversation is not a string of unrelated questions; it is a thread, and half of what any turn means lives in the turns just before it. Say "and the second one?" and the fast reply has nowhere to look — the thing "the second one" points back to lived in the previous message, which was answered, sent, and a beat later filed as its own diary fact, but is not in front of this reply as conversation. Lexical diary search will not reliably close that gap either: the referent is a pronoun, carrying none of the words that would surface the fact it stands for. The reply needs the recent exchange itself, held the way any conversation partner holds the last few things said.

So the fast pass should also carry a short rolling window of the last N messages as working memory — kept verbatim while the window is small, and summarised down to fit when it grows past the model's budget, through the very same context-budget guard every generative call already passes through. This is a third kind of memory, and worth naming apart from the two already standing. The diary retrieval is long-term recall — everything ever filed, fetched by relevance. The deep second pass carries an origin reference so a late enrichment can situate itself against everything said since it was set in motion. This new one is neither: it is the short-term conversational thread the fast answer needs to stay coherent from one turn to the next. It fell out of the first cut of the read path for an understandable reason — the fast reply was proven against single messages, and against both an empty and a populated diary, which is precisely the shape against which a stateless reply looks whole. A live back-and-forth is where the absence shows, and a live back-and-forth is exactly what the shell now drives against. It goes down as an undone item on the read path's Tier 1, to build alongside the deep pass rather than after it.

## what comes next: building Tier 1 short-term conversational memory

That undone item is where this session goes next. Tier 1 — the short-term conversational memory named just above — is still unbuilt, and it is the brick we pick up now. The gap is already diagnosed and the shape already argued: a short rolling window of the last few turns, carried into the fast pass as working memory, held verbatim while it is small and summarised down through the existing context-budget guard once it outgrows the model's room. What remains is to actually lay it — to give the fast reply the recent thread it has been answering without, so that "and the second one?" has somewhere to look. The diagnosis was the easy half; the build is the point of the next stretch of work.

## before the brick: writing the two paths down, and tightening the code

Before picking up that brick I stopped and did something the loop had been running without: I wrote two of its paths down. The read path first, because the gap named just above only reads as a gap once you can see plainly what the fast reply *is* allowed to know — and nothing said that on paper. `doc/context-engineering.md` says it now: the persona, the single message, and the lexically-relevant diary facts, three inputs and only three, traced from the worker's claim through the librarian's blended English-and-French rank down to the summarisable slice the budget guard would condense first, with the anonymous caller's dead-end drawn in so the boundary shows. Reading it back, the short-term hole is exactly where the page goes quiet — which is the point.

Then auth, which had quietly grown two invariants worth stating outright rather than leaving implicit in the code: nothing here is an oracle — `/login` and `/login/verify` answer identically whether or not an address is registered and whether or not a guess is right — and every abuse limit is held by the shape of a database row, not by the order requests happen to arrive. `doc/authentication.md` draws both, the handshake as a sequence diagram and the attempt budget as a flowchart, and it exists because of a question I couldn't answer straight from memory: can a wrong code be guessed at forever? It can't. `verify_login_code` charges each wrong guess against the code's own `failed_attempts`, and the single statement that increments the counter also burns the code the instant it reaches the budget — so brute force is a fixed number of tries against one code, not a race decided by how fast an attacker can hammer.

Writing that down pushed two small changes into the code itself. The login code went from six digits to eight — still one glance to type, a hundredfold more room to guess against within the same ten-minute life and the same budget — and the test that recovers a code from a sent email now derives its digit count from identity's own constant, so the regex and the code length can never drift apart the next time that number moves. And I pulled the budget suggested in the example env from five down to three, on the plain reasoning that three wrong guesses is already well past a fingers-slipped mistype and comfortably short of anything worth calling an attempt at the code; the enforced default in `config.py` still reads five, so for now that tightening is a recommendation to whoever deploys rather than a change to what the kernel does out of the box.

## the shape the brick took: a memory gradient, not a count of messages

Sitting down to design that rolling window in earnest turned it into something better than the sketch it started as. The first framing carried the right instinct on the wrong unit — *the last N messages*. A count is blunt in both directions at once: it token-bombs a fast burst of one-line exchanges and starves a slow, dense one, and whatever number you pick is wrong the moment the rhythm of the conversation changes. What the quick reply actually wants is not a fixed number of turns but a *gradient* — the recent exchange held word-for-word, everything older folded into a running summary that never stops existing. Near things sharp, far things blurred but never lost: the shape memory already has.

So it settled into two buckets. The present is the recent turns carried verbatim, both sides of the exchange interleaved exactly as they were said — the machine and the human, and the machine's own unprompted missives too, because the thread is the whole relationship and not just the half the human starts. The past is a single running summary, the Gist, holding everything that has aged out of the verbatim window. The reply reads both: the Gist first for the backstory, then the verbatim tail for the live exchange it is the next line of.

The trick that keeps it fast is doing the arithmetic at write time. Every utterance's token weight is measured once, with the same counter the budget guard already uses, and stored the moment the line is written — off to the side, never on the intake record, which by its own rule keeps nothing derived from the words beside them. That stored weight is what the background fold reads to know when a tail has grown large enough to trim. The size budget is a reserved slice of the active model's context window, so a fallback down the ladder carries its own budget with it.

I first drew the read itself as a token wall — Postgres walking the turns newest-first, stopping the instant the running sum crossed that slice. Correcting that was the point of this session, and it matters enough to say plainly why. A capped read plus an asynchronous fold leaves a seam: turns that have fallen past the cap but that the background worker hasn't yet folded into the Gist sit in neither place — out of the verbatim tail, not yet in the summary. A fast reply built on a memory with a hole in it is just a fast way to make something up. So the read no longer caps. It carries every turn back to the exact point the Gist ends, however many tokens that is, and the two touch with no gap. The budget stops being a wall on the read and becomes purely the *trigger* for the fold: when the tail grows past it, the worker folds the oldest turns forward and the tail shrinks back. If the worker lags, the only cost is a temporarily fatter prompt. State consistency over a few saved tokens: the buckets meet, always.

That left one loose end worth pulling on. With the read uncapped, a badly lagging fold could in principle grow the prompt past the window the model reads well — and the old guard, faced with that, would shrink the diary facts and, if there were none, simply give up and raise. Raising is going blind by another door. So the guard now answers to one clean rule for the whole prompt: two things are sacred and never shrink — the persona (who is speaking) and the live input (what was just said, or the agent's own line) — and so are the little instructions that tell the model how to behave. Everything the reply *remembers* — the diary and the whole conversation — is one block the guard may squeeze to fit. So an overrun is met by condensing what is remembered, never by dropping who it is or what it was just asked, and a reply always composes. In the ordinary case nothing is squeezed at all; it is a backstop, not a step.

The Gist is append-only, and I trust it more the longer I look at it. Each time the background sweep folds the overflowed turns in, it does not overwrite the summary — it writes a *new* row carrying the freshly merged paragraph and a hard integer marking the last turn it absorbed. The current Gist is simply the newest row; every version it ever held stays behind it, an inspectable history of how the memory grew. The cutoff is a real foreign key the code reads straight, never a number parsed back out of the summary prose — a fact the schema states exactly rather than one a model has to guess at. And because every fold re-compresses the whole paragraph down to a fixed target and truncates to it, the Gist cannot creep past its size no matter how long the conversation runs. It works on its own thread, off the path the human waits on, and folds each turn exactly once, because the cutoff only ever moves forward.

None of this disturbs what already stands. The lexical diary reach, the reply composition, the live ingestion — all shipped, all proven, all untouched. Short-term memory is added *beside* long-term memory, a second kind of recall folded into the same prompt: the diary answers "what do I know that bears on this?" by relevance, the conversation answers "what were we just saying?" by recency. The only thing the two share is the token window, split between them by fixed reserved slices. Before laying a line of it I wrote the whole design into the read-path document, diagrams and all, so the build answers to a spec rather than to a memory I would otherwise have to reconstruct.

## the brick, laid: the gradient built and green

The spec is code now. A migration adds the two tables the design named — the stream of utterances and the append-only Gist — and the stream carries the constraint I most wanted the database to hold rather than the code to remember: a row points at exactly one source, an intake line or a missive, never neither and never both, enforced by a check the schema states in one expression. The token count lives on the stream row, computed at write, exactly where it is allowed to live and nowhere the intake record forbids it.

The logic sits in one new module, and it reads the way the diary's librarian reads — a thin, honest data layer with the cleverness pushed into the SQL. The verbatim tail is simply every turn newer than where the Gist ends, in order, resolving each turn's words back through its pointer; it drops the very message being answered, because that line was written onto the stream the instant it arrived and would otherwise show up twice — once as the last thing said, once as the thing being said now. The fold's own reads are what use the budget: they find a tail that has grown past it and hand the oldest of those turns to the summariser with the cutoff they reach, leaving the recent ones verbatim. The fold hands those turns to the summariser and truncates its own output so the summary's size is a promise and not a hope.

The three places an utterance is born now each add their line to the stream in the same breath as the durable write they already made: the intake route when a recognised human speaks, the worker when the machine's reply is marked answered, the missive delivery when the machine reaches out unprompted — every one of them transactional with the record it mirrors, so the stream can never drift from the truth it points at. And a fourth background loop joins the three that were already there, folding the overflow into the Gist on its own thread, gated by its own switch, sharing nothing with the others but the signal that winds the whole process down.

It is all under test and green — the whole suite, the new coverage included: the write's token counting and its one-source rule, the read carrying the whole tail back to the cutoff with no token cap, the fold trigger firing only when the tail has grown past its budget, and the cutoff that keeps a folded turn from ever folding twice. The last piece is a deploy backfill: a script that reconstructs the stream from the history already on the box, in the order it happened, counting each old utterance's tokens the same way the live path will — idempotent, and a dry run until you tell it otherwise, so the first reply after this ships already sits inside the conversation that came before it.

One choice I had made on instinct did not survive contact. I had the fold lean on the fast local model, off the paid path, on the reasoning that a summary no one waits on has no business costing a metered cloud call. The by-hand smoke against a live box put the lie to it: asked to boil four short turns down, the little model wrote a bloated paragraph and slipped a garbled clause into it — a line claiming the baking blog lacked what it needed to generate weather data, which is simply not a thing anyone said. The Gist is not a place that tolerates either fault. It is the whole of what a later reply reaches back through once a turn has aged out of the verbatim tail, so a fold that bloats or garbles does not fail loudly — it quietly degrades every reply that leans on it downstream. The saving is not worth that. The fold now runs the same heavy model that composes the replies; it is still background work no one waits on, and it fires only when a tail overflows, so it is a long way from every turn — the metered call buys a Gist worth trusting, and the one-line knob to point it back at the local tier stays there for the day the cost ever outweighs the quality. Under the heavy model the same four turns fold to a third of the words, every fact intact and nothing invented — the summary a later reply can actually stand on.

The rerun taught a second, subtler thing, and it was the probe that was wrong, not the memory. The smoke first asked "how was I feeling about the second one?" — and the reply answered about the second *time* the human nearly gave up, not the second *project*, which was what I meant. I had read that as a miss; it is not. The question is genuinely ambiguous — there are two pairs of things in the conversation "second" could point at, the two projects and the two near-quits — and the reply reached for the nearer one, which sat word-for-word in the verbatim tail, over the one summarised up in the Gist. Recency out-pulled the summary, which is exactly the weighting I want: the sharp, present turns should win a tie against the blurred past. What that exposed was a soft test, not a soft memory — a probe whose answer sat in the tail proves nothing about whether the Gist can be reached at all. So I rewrote it to ask the one thing the tail can no longer answer: which project had the forecast trouble — a fact that had already been folded out of the verbatim turns and now lived *only* in the Gist. The reply named the weather app and the forecasts that fought it. That is the proof the first probe never gave: the fast answer reaching up into the summarised past for something the present no longer holds.

## one fold at a time: closing the race on the summariser

With the gradient built and the fold proven to produce a Gist worth trusting, one property it did not yet hold was safety under more than one worker. The fold reads which symbiot has overflowed, lets the connection go for the length of the metered model call, then writes the new Gist on a fresh connection — and I had rested exactly-once on a single fact: the cutoff only ever moves forward, so a turn folded once falls behind the boundary and is never gathered again. That reasoning is airtight across passes taken *one after another*. It says nothing about two folds in flight *at the same instant*. Run two sweeps — which is precisely what a multi-worker deployment is — and both can read the same over-budget tail at the same cutoff, both spend their five seconds summarising, and both append a Gist row for the identical boundary. Two summaries of the one span, a fork where there must be a single line. This is the same lesson the intake queue already learned and wrote down: the invariant has to live in the database, not in the hope that the timing stays polite. So I laid the brick now rather than filing it as a someday-if-we-scale note — the correctness is buildable today, and the sole human this runs for deserves it as much as a thousand would.

The fix is a lock, but a carefully chosen one. The obvious move — lock the symbiot's own row while the fold runs — would have been a quiet mistake: that same row is touched by the proof-of-life re-arm on every inbound message, so a fold holding it would stall the very path a human waits on, coupling a background chore to the live conversation. Instead the sweep takes a Postgres advisory lock in its own private namespace, keyed on the symbiot and meaning nothing but "this symbiot's fold is mine right now." Nothing else in the system ever reaches for that lock, so it can never block a reply or an inbound line — and the read path stays exactly as lock-free as it was, which was the one line I would not cross. It is the *try* form: a second worker that finds the lock held does not queue behind the first's model call, it gets a plain no and moves on, leaving the symbiot eligible for the next pass. And it is scoped to the transaction, not the session, on purpose — it releases itself the instant the fold commits or rolls back, so a crash mid-summary strands nothing on a pooled connection. The one price is that the whole fold now runs inside a single held transaction rather than handing the connection back during the model call; with a single sweep thread per process that is one pooled slot occupied briefly, never the reader's, and a fair trade for a summariser that can no longer fork. A test stands over it now: while one worker holds the fold, a second sweep across the same overflow claims nothing, spends nothing on the model, and writes no second Gist — then, the lock released, the fold goes through exactly once.

## the fold reads its own output, so it must be spotless: closing the yapping bleed

One last property of the fold had been sitting in plain sight, and it is peculiar to this store in a way that made it worth stopping for. The Gist is the one place in the kernel that reads its own past output back in: every fold takes the *current* Gist as part of its input and writes a fresh one, so the summary is a loop feeding itself. That closes a door most of the other model calls leave open harmlessly. When a model, asked to summarise, wraps its answer in the little pleasantries models reach for — "Here is the updated summary:", a stray ``` fence, a "Sure," before the prose — that filler does not just clutter one row here. It becomes input to the next fold, which faithfully re-summarises it, which the fold after that re-summarises again. Meta-text that slips in once does not wash out; it compounds into the anchor and stays. The prompt had asked, politely, to "return only the merged summary, nothing else" — but that is a plea a model is free to ignore, and against a store that recycles its own output, a plea is not a guarantee.

The kernel already owned the right answer and was using it everywhere else. The ontology's LLM calls do not hand the model a blank page and hope; they hand it a Pydantic schema that becomes the decoder's output grammar and is re-validated on the way back, so the reply that crosses the boundary is a typed object with its fields already checked. The fold was the odd one out, still generating free text. So I moved it onto the same boundary: the merge now returns a `_FoldReply` — a model with a single `summary` field — through `generate_json`. The summary is the only field the model can fill, which is the whole point. A preamble has no field to land in, a code fence has nowhere to go; the structure makes the filler impossible rather than asking the model not to produce it. The field carries a `min_length` so an empty summary fails at the boundary too, the same way the concept extractor refuses a fact that is about nothing.

I proved it from both ends. A pair of unit tests pins the shape — the fold reaches for the schema path and never the free-text one, and an empty summary is refused — so a later hand cannot quietly revert it to `generate` without a red bar. Then the by-hand smoke against the live heavy model, which is the only place the real thing's raw output is ever seen: it folds a real exchange and now checks the Gist opens straight into prose, no fence and no preamble, before it trusts it. Run live, the summary came back clean — pure past-tense prose, every fact intact — and the follow-up that can only be answered from the Gist still named the weather app and the forecasts that fought it. The read-path document gained the reasoning under a "why the Gist stays clean" heading, set beside "why the Gist stays bounded," because the two are the same kind of promise: one guards what the anchor may grow to, the other guards what may reach it at all — and both are held by structure, not by asking the model nicely.

## the deep second pass: reaching by meaning, and following up only when it earns the interruption

The two things Tier 2 had been waiting on both landed — the ontology write path that gives it structured facts to reach through, and the heavy model on Scaleway that its judgement leans on — so the deep second pass was finally a brick I could pick up rather than a note filed behind a blocker. It is the other half of the read path, and it answers a different question from the fast reply entirely. Tier 1 answers *now*, on the path the human waits on, by matching the words of the message against the diary. Tier 2 answers *a beat later*, off that path, by matching its *meaning* — and it says nothing at all unless the reaching turns up something the fast answer missed. The shape it takes is a fifth background sweep, cut from the same cloth as the three already running: once a message's fast reply is settled, the sweep reaches deep, decides whether there is anything worth adding, and if there is, follows up.

The reaching itself is two movements, and I kept them in their own module, well away from the lexical librarian — the fast reach is the thing the deep pass exists to *complement*, not to touch. The first movement is vector recall: the message is embedded and the nearest diary facts are pulled by cosine distance, so a fact surfaces because it *means* something close to what was said, even when it shares not one word — "I can't seem to recharge" reaching an entry about staying at the office past midnight, which the lexical pass, seeing no common term, could never find. The second movement walks out from those recalled facts along the ontology the write path grew: it reads the concepts each recalled fact is filed under and pulls the *sibling* facts filed under the same concepts, so the reach follows the vocabulary tree, not raw distance alone. The one line I would not cross there: the walk *reads* the ontology and never writes it. It would have been easy to let the pass route the message through the router and mint whatever new concepts it named — but minting vocabulary from a *read* is the write path's job, and a question quietly growing the diary's tree every time it is asked is exactly the kind of coupling that corrupts a store slowly. So the deep reach takes no lock and makes no write; it only ever queries.

What comes back from that reach is handed to the machine with the whole of its own recent past, because a follow-up that lands after the conversation has moved on has to know where things now stand. That is the origin reference, and it carries three things: the message that prompted the fast answer, the fast answer itself, and every turn exchanged since. The middle one is the one I most wanted in place, and it does two jobs at once. It is how the gate judges whether the deep facts add anything — you cannot tell new ground from old without seeing the ground already covered — and it is what keeps the follow-up from simply paraphrasing what was already said. Having the prior answer sitting in the prompt is necessary but not sufficient; a model handed its own words will cheerfully restate them in a fresh coat, so the compose instruction names the earlier answer outright and forbids repeating it. No-repeat is spec'd, not left to the model's better nature — the same discipline the fast reply already keeps when it is told to say nothing the memory does not support.

One structured call both gates and composes. It reads the persona, the origin reference, and the deep facts, and returns whether to surface at all and, if so, the follow-up's words — a typed reply through the same strict-schema boundary the router's calls cross, so the decision and the prose come back checked rather than parsed out of free text. When the deep reach found nothing, the call is skipped entirely: there is no new ground to weigh, so no metered model call is spent deciding there was nothing to decide. And the bar to surface is set high on purpose. Silence is the right answer more often than not — the machine already answered, and a second unprompted message has to be worth the interruption — so a pass that adds nothing records itself as considered-and-silent and sends nothing. A follow-up that does clear the bar goes out as a *missive*, the kernel reaching out on its own, discovered through the inbox on its own terms, rather than pretending to be an inline reply to a message the conversation has already left behind. It flows through the delivery that was already built for the machine's unprompted lines — recorded durably, mirrored onto the conversation stream so the next turn remembers it, nudged over two channels so first contact never rides on one holding up.

Exactly-once is pinned the way the rest of the kernel pins it — in the schema, not in the sweep being careful. A new provenance table carries one row per message the pass has *considered*, surfaced or not, under a unique link back to the message; the sweep's eligibility is the mirror of that link, an answered message with no row yet. Answered specifically, narrower than ingestion's net: ingestion files an abandoned message too, because its words are still the diary's, but there is nothing to *enrich* about an answer that was never given. A crash mid-pass leaves no row and the message simply eligible again; a committed pass records its verdict and is never re-run. And because the whole pass — the deep reach, the composing call, the send — is several seconds of work that reads state before it writes, it carries the same race the summariser fold did: two sweeps could both spot the same eligible message, both spend the model call, and both deliver a follow-up for the one message. So it takes the same cure, a non-blocking advisory lock in its own private namespace keyed on the message, claimed before the heavy work; a second worker that finds it held gets a plain no and moves on. The follow-up and the row that records it commit together, so a message is sent-and-recorded atomically or neither.

It is all under test and green — the eligibility and the answered-only narrowing, the claim that makes a second sweep skip, the origin reference gathering exactly the turns said since, the gate that suppresses without a model call when the reach is empty and downgrades a yes-with-no-words to a silence, and the ontology walk pulling siblings by shared concept most-shared-first. The vectors themselves, which need the live embedding model, went into a by-hand smoke that stages the whole reason Tier 2 exists: a diary filed full of entries circling one theme in words the message never uses, a message about that same theme in different words, and a deliberately diary-blind fast answer that missed the connection. Run live, the lexical reach comes back thin and the meaning-based reach surfaces the themed facts the words could never reach — the contrast the fast pass cannot close on its own, closed a beat later, off to the side.

Running it live taught the last thing the faked tests could not: that the gate actually *discriminates*, and that when it decides to speak the missive really goes out. On the first message — "I can't seem to recharge" — the gate chose silence, and it was not wrong to: the fast answer had already invited the human to say more, so the depletion facts the deep reach dug up added little the exchange wasn't already heading toward. The proof the trigger is live came from a second, sharper case: asked whether to take on leading a new project — "I think I'm ready" — with the same overwork and lost-sleep facts on record, the gate surfaced, and the follow-up it wrote named the cause outright, warned that a readiness coming from an already-overdrawn baseline is a contaminated signal, and told the human to sleep before deciding — reframing the bland "sounds like a great step" rather than repeating a word of it. That is the whole promise made concrete: the machine is *able* to reach back, decide, and follow up unprompted, and it *does* when the reaching earns the interruption — silent when the answer already sufficed, speaking when there is something the human needs said. The send itself is not a matter of the model's mood: once the gate returns a follow-up, the sweep raises the missive, mirrors it onto the stream, and nudges — deterministically, the same delivery every other missive rides. The only judgement is whether to speak; the speaking, once judged, is certain.

## the rung, closed

The deep pass went in as one changeset, and with it the read path is whole: the deep-reach module and the enrichment orchestration, the provenance table that pins the pass exactly-once, the background sweep and the thread that runs it, the config knobs it answers to, the read-path document brought level with the code, and the two layers of tests — the faked suite and the by-hand live smoke — all landed together. Two small tidyings followed the commit: the symbols in both new modules were put in alphabetical order — the deep reach's three functions and the enrichment module's private helpers — so the files read the way the rest of the kernel does. Everything green through it.

That closes the read-path rung outright. Its four workstreams all stand and are proven: the fast lexical reply in the machine's own voice, the short-term conversational memory that keeps a thread coherent turn to turn, the live ingestion that fills the diary as messages arrive, and now the deep second pass that reaches by meaning and follows up only when it earns the interruption. The one deploy step the rung still waited on — the one-off backfill that reconstructs the conversation stream from the history already on the box — has been run there for real, so the first reply after this sits inside the conversation that came before it. The diary is read from both ends now, fast and deep, and the loop the whole rung was arguing for runs end to end, live.

## the next line to cross: the loop learns to act, and a reminder is the first tool

With the read path whole, a thing becomes plain that the read path itself could never expose: everything the loop does is *words*. It answers, it follows up, it reaches back through the diary and up through the Gist — but every one of those is speech. It has never once reached *out* and changed something in the world on the symbiot's behalf. That is the line the next stretch of work crosses, and it is a bigger line than any inside the read path, because it is the difference between a thing that talks and a thing that acts.

The smallest honest place to cross it turns out to be the humblest thing the old OpenClaw setup already did for me: hear "remind me of this at that time," and at that time, say it. A one-shot reminder is the right first action for a reason that has nothing to do with reminders being important and everything to do with them being *clean* — it needs no external driver, no third-party credential, no calendar to reconcile. It writes a row to our own store and fires over the reply path already built. So the thing actually under test is the machinery of acting, not the plumbing of some integration; and its pass/fail is unarguable — did the right words fire at the right moment, exactly once. You can drive that end to end and know.

So the question I sat with was not whether to build the reminder but what "first tool" should mean. There were two honest readings. One is bespoke: a reminder-only extraction pass that recognises the intent, pulls out the message and the time, and files it — smallest possible, but it is not really a *tool*, it is a special case wearing the word. The other is foundational: build the actual tool-calling boundary now — a registry where a tool is a name, a Pydantic argument schema, a code executor, and a result handed back to the model — and populate it with exactly one tool. I went with the foundational read, and the reason is the "lay the brick now" discipline this build keeps everywhere else: the registry is the correct shape, it is buildable today, and it does not balloon scope, because only one tool exists inside it. The mechanism and its first inhabitant are born together, correctly, and the second tool is then just a new entry rather than a rewrite.

What makes that cheap rather than grand is that the boundary is one the kernel already owns and already trusts. Every structured model call here — the fold that keeps the Gist clean, the ontology's concept calls, the deep pass's gate — hands the model a Pydantic schema that becomes the decoder's grammar and is re-validated on the way back, so what crosses the boundary is a checked, typed object rather than free text parsed on faith. A tool call is exactly that shape, pointed at an *action* instead of a *summary*: the model decides whether to call and emits validated arguments, and then code — never the model — carries the effect out. This matters as a matter of principle and not just tidiness. The loop is about to be able to *do* things, and the one invariant I will not cross is that the model's reach ends at deciding and describing; the doing stays in code the whole way. So the agent's first step toward acting is not a new, looser surface where the model is suddenly trusted with the world — it is the strict boundary we already lean on, aimed for the first time at doing rather than saying. That framing is written into the issue that carries the work, so the build answers to it rather than to my memory of this evening.

## the shape the tool call takes: retrieve, decide, act, speak

With the principle settled, this session went down into the mechanism itself, and the first thing I fixed was what the tool-calling boundary would *not* be. Every provider we lean on — Scaleway, Mistral, Ollama — ships its own native function-calling API, and the obvious move is to use it. I turned it down, and the reason is durability, not taste. Those APIs are provider surfaces, and provider surfaces churn; binding the kernel's internals to one of them makes the loop hostage to a shape we don't own and can't hold still. And we don't need it: native function calling is, underneath, the same constrained decoding the kernel already does everywhere through `generate_json` — a strict Pydantic schema that becomes the decoder's grammar and is re-validated on the way back. So tool-calling rides that boundary we already own and trust. The decision of *which action* is expressed in our own shape; only the thin per-tier call layer ever touches a provider's specifics. We reach for a provider's native tool API the day the kernel needs what it is actually for — many tools chained in a multi-step loop — which is a long way past this first one.

The next thing that fell into place is that tools want to be recalled by meaning, exactly the way the diary is. A tool is a name, a description, an argument schema, and a code executor — and those don't all live in one place. The first three, the descriptor, live in the store as a searchable row with an embedding of the description; the executor is code, in a registry keyed by that name. The store is the index you search, the code registry is the dispatch table you land on, and the name is the join. That split is what makes "code executes, never the model" structural rather than a hope: the model can only ever produce a name, and a name resolves to a callable we wrote — an effect can never be a thing the model emits. The code registry is the source of truth for which tools exist; the catalog is derived from it and reconciled on startup, never hand-edited.

Then the flow, which came out cleaner than I first drew it. My first sketch had a gate call — "should I use a tool at all?" — sitting in front of a tool search. Talking it through, the two collapsed into one: let the *search* be the gate. The message is embedded and matched against the tool catalog before anything is composed, and when nothing clears the relevance bar, the message is ordinary and takes the ordinary reply path untouched — no decision spent, no machinery in the way, which is the case for almost every message. Only when the search surfaces candidates does a decision call happen at all. That search is coarse recall on purpose; its job is to not miss a candidate, and the precise judgment comes next — the same two-stage shape the ontology re-ranker already uses, vector recall proposing and the model rejecting.

The decision call itself I kept deliberately light: it sees the shortlist and the recent conversation tail, and not the full diary. The tail earns its place because arguments refer back — "remind me about *that* at six" only resolves against what was just said — so a decision blind to the thread would route correctly and then extract a useless argument. The diary stays out because this call is for routing and argument extraction, not for composing a rich answer. And that is the reason the decision and the reply are two calls rather than one: the decision stays cheap on every message that merely sits near a tool, and the reply stays memory-rich when it is actually a reply. When the decision comes back "none" — a candidate surfaced but nothing truly fit — it hands off to the ordinary reply, which has the memory to answer well. The cost of the split is honest and small: a message that looked tool-shaped but wasn't spends two calls instead of one, on the narrow set of messages that sit near a tool without invoking one. Worth it.

Two properties I would not leave to chance closed the design. The executor's write runs on the worker's own thread, in its own transaction — never inside the killable child, because the child can be severed at the deadline mid-run and a half-done effect is exactly the kind of thing the rest of the kernel is careful never to leave behind. So the shape is: the child decides, the worker thread executes, the child composes the confirmation. And the effect is exactly-once against the message that triggered it, guarded in the database the way enrichment already guards itself — because a message can be retried, and a first run that fired the effect and then failed on the confirmation call must not fire it again. On a retry the write conflicts and is a no-op, the effect already stands, and only the spoken confirmation is re-derived. Whichever way the flow goes — tool fired, or a plain reply down either off-ramp — it always ends in something said back to the human; no silent action, no dead end, the same promise the read path already keeps.

The whole of it went into `doc/tool-calling.md` before a line of it is built, the same way the memory gradient's design went into the read-path document ahead of its build — so the code answers to a written spec rather than to my recollection of the evening it was argued out.
