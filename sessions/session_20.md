# session 20 of building The Joy in the open

## what this session is about: the duplicates lazy minting can't help but breed

The last session got a fact travelling the whole path — named into its concepts, each one recalled, scored, and matched-or-minted, the fact written to the store against every type it earned. The vocabulary now grows from use: a genuinely new kind of thing coins a type, a kind already seen reuses one. That is the design, and it is working. This session opens on the flaw baked into that very design — not a bug in what was written, but a consequence of how it was written — because it is better to name it plainly, and understand exactly how far the guard already in place reaches, before laying a brick against it.

## the flaw: forward-only minting breeds semantic duplicates

Lazy minting only ever moves forward. A concept arrives, the router looks at the store it has *now*, and either reuses a type it finds or coins a new one — and once coined, a type is committed and never looked at again by later mints. Nothing ever goes back to notice that two types it minted on two different days are really the same idea. That is what "forward-only" means, and it is why the ontology will, left alone, slowly fill with semantic duplicates: two rows with different names that mean the same thing — `ProgrammingLanguage` and `CodingLanguage`, `Car` and `Automobile`, `partner` and `training_partner` — each coined in good faith on a day the other wasn't surfaced.

It is worth being precise about why the machinery *can't* prevent this on its own, because the reason isn't a missing check — it's the nature of the decision. Whether an incoming concept reuses an existing type or coins a fresh one rests on two soft steps in a row: recall has to surface the near-synonym among the nearest vectors in the first place, and then the re-ranker and the band have to score its fit above the reuse floor. Both are judgments, not lookups. Recall can miss a true synonym whose definition happened to embed a little too far away; a real match can land in the grey zone or just under the bar and get read as "nothing fits well enough — coin." Every one of those near-misses mints a new name for an idea the store already held, and because minting never looks backward, the twin sits there permanently. The reuse threshold — the single knob the whole growth law turns on — sets *how often* this happens, and it can be tuned to make it rarer, but there is no setting that makes it never happen. Over enough facts, the near-twins accumulate. That is the "inevitably" in the ticket, and it is honest: it is a property of routing by similarity, not a defect we forgot to fix.

## what the guard already in place does — and doesn't — catch

There is a guard on minting already, and it is easy to mistake it for a fix for this. It isn't, and the difference is the whole point. When the minter coins a name that a past fact already coined, the store refuses the second row — types can't share a name — and rather than manufacture a suffixed `_2` twin, the clash is read as *the store already has this* and the fact links to the existing type instead. Two mints racing to coin the same name at once are collapsed the same way, by `ON CONFLICT` in the insert, so the database picks one winner and the loser reuses it.

That guard is real, but it keys on the **name string being identical**. It collapses exact-name collisions — the same label coined twice — and nothing more. Semantic duplicates are the opposite case: the names are *different*, so to the unique-name rule they are two perfectly distinct types, and both are written and both persist. The dedup backstop never fires, because from where it sits there is nothing to dedup. So the two problems sit cleanly apart: exact-name collision is closed, in the database, by construction; semantic duplication is wide open, and by the argument above it cannot be closed on the write path at all — a name that doesn't yet exist is always a legal thing to coin.

Which points at the only place it *can* be closed: not forward, on the way in, but backward — a reconciliation pass that reads the store as it stands, finds types that have drifted into meaning the same thing, and merges the near-twins into one, re-pointing every fact that referenced the losers. The write path keeps doing what it does; a separate, later sweep keeps the vocabulary from silting up. That merge pass is the brick this session is here to reason about and lay — but before reaching for it, it's worth being sure the mess is even worth cleaning, because the answer turns out to be sharper and more mechanical than "duplicates are untidy."

## why duplicates actually hurt: retrieval runs on a fixed budget

It's worth being concrete about *why* a store full of duplicates is bad, because it isn't obvious at first — no fact is ever lost, and each one is filed under a true type. The harm is entirely on the way back **out**, when the store is read to answer the symbiot, and it comes down to one hard limit: a retrieval hands back a fixed number of results, and duplicates spend that budget on saying the same thing over and over.

Picture the read path the way last session sketched it. When the human asks something, a fast, plain **text search** answers straight away, so a reply lands without making them wait. Then, quietly in the background, a slower **search by meaning** runs over the vectors, and *if* it turns up something the quick answer missed, a second, richer reply can follow a beat later — an optional deepening the symbiot receives after the fact, not a delay they sit through. That background pass is the one that leans on the vocabulary and the embeddings, and it is exactly the one duplicates cripple.

This mechanism is **Top-K starvation**. The meaning-search pulls back the N nearest facts — say the closest twenty — and only those twenty ever reach the model. If fifteen of them are near-identical ("I live in Strasbourg," told fifteen times across three years), they physically crowd out the fifteen *different* facts that would have made the deeper answer worth sending. Cleaning up *after* the search doesn't save it: collapse those fifteen down to one and you're left with six facts where there should have been twenty. The window was already spent before any cleanup could run — the duplicates blinded the search itself, upstream of every filter.

The same crowding happens one level up, in the vocabulary. Duplicate *types* fracture "everything of a kind": ask "how's my training been going," and if training was filed under three synonymous types over the months, you get back only the slice that happened to land under the name the search picked — silently, with no hint the rest exists. And it feeds itself, because duplicate types also crowd the *write* path's candidate pool, so new facts route worse and mint yet more duplicates. Left alone, the mess grows on its own.

So the tempting fix — clean it all up at read time, clustering and deduping and sorting the results on every single question — is exactly the wrong place to put the work. That drops the heaviest possible cost at the one moment the human is actually waiting, and it gets slower as the diary grows; a diary that has to think for ten seconds before every answer is a broken diary. The work per question has to stay roughly flat no matter how large the store gets. Which means the store must be kept **clean and resolvable ahead of read time**, so reads stay cheap — and that splits into two jobs, one for each kind of duplication:

- Redundant **facts** about a stable state ("live in Strasbourg" ×15) are not something to merge away — every one of them is true, told on a real day. Instead each fact carries cheap timestamps, so read time can collapse them to the *current* state with a plain sort by time rather than an expensive clustering job. That's the two clocks below.
- Duplicate **types** (`workout_action` ≈ `training_session`) genuinely are one idea wearing two names, so those we do merge — off to the side, on a schedule, never on the read path — so the vocabulary and the recall window stop being wasted on synonyms. That's the offline pass further down.

The through-line, and the reason the design accepts a messy write and tidies up afterward instead of chasing a perfect write: keep the raw facts whole, like an event log that's never rewritten, but keep the *derived* structure — the live types, the current state — resolved and tidy, so the read path skims a clean surface instead of digging through the raw pile on every question.

## two clocks on every fact: when it happened, when it was told

The analysis above leans on the store knowing *when* each fact sits on the timeline; so far it knew only one time, and this session gives it the other. Every filed fact already carried `created_at` — the moment the row was written, which for live ingestion is the moment the symbiot told us the fact. That is the telling clock. What it lacked is the event clock: `happened_at`, when the thing actually occurred in the world. A boxing session reported tonight but fought yesterday was recorded tonight yet happened yesterday, and the diary should know the difference. Two clocks, then — when we heard it, and when it happened — because a life doesn't record itself in the order it's lived. We weighed a third, a `reported_at` split apart from `created_at` to mark the telling separately from the row's birth, and let it go: for live ingestion the two are the same instant, and the one day they'd diverge — backfilling or replaying an old entry — isn't here, so a column for it now would be structure bought for a future that may never arrive. `created_at` carries the telling; `happened_at` carries the event; that is enough.

`happened_at` is nullable on purpose, and the fallback is the whole point of letting it be null. Plenty of facts arrive with no temporal cue at all — a bare "I live in Strasbourg" points at no moment in particular — and rather than guess a time and pretend to a precision we don't have, we leave the column empty and let it **collapse to `created_at`** at read time. So the rule is simple: use the event time when the fact gave us one, otherwise stand in the time we recorded it. No fact is ever without a usable timestamp, and none is ever fitted with a fabricated one.

There's a scope line under this worth stating plainly, because the ingestion design spent real effort refusing exactly this kind of thing. The write path is deliberately thin: a fact is filed as its *kinds* and its raw text, and the particulars inside it — the "Jeremy," the "45 minutes" — are left in the source line, never pulled out into named payload fields, because free-invented field names breed the same duplication the type vocabulary exists to kill. Time is the one particular we promote out of the raw text and into first-class structure, and it survives that exception cleanly for a specific reason: it is a **fixed schema column, not a coined key.** There is exactly one `happened_at`, hardcoded, with no room for the model to name the same idea `date` one day and `when` the next. Every other particular stays where it lives. Event time earns the promotion because nearly every question a diary is ever asked is a question about time — *what did I do last week, when did that start, how long since* — and a store that couldn't order its own facts on the world's clock would be answering all of them blind.

## what gets built, and in what order

Two bricks fall out of all this — one for each kind of duplication the retrieval section split apart — and the order they go in matters. The two clocks come first, because they're the foundation everything about read-time cleanliness stands on, and because they close the *fact-level* half of the problem: the "I live in Strasbourg" ×15 that starves the retrieval window is tamed the moment every fact can be sorted by time and collapsed to its current state. Laying them is a one-column schema change plus a small write-path step that reads a time out of the raw fact when one is there — the temporal extraction. Every promise about resolving state cheaply at read time is empty until the facts actually carry their times, so this lands before anything is built on top of it; it is also the more invasive change — a migration, and a new step on the write path — and it's cleaner to settle before setting the merge pass down beside it.

The merge pass — the "backward" reconciliation named at the top of the session — is the second brick, and it closes the *type-level* half: the `workout_action` ≈ `training_session` synonyms that fracture "everything of a kind." It's worth being honest that it doesn't technically *depend* on the clocks — it collapses duplicate types, which needs no fact times at all — so it goes second because building on a settled foundation is cleaner, not because one blocks the other. In plain terms, here is the shape it will take, which is also the answer to the questions left open earlier — what counts as "the same," who survives, how often it runs:

- it runs off to the side on a cadence, **inside the app** the way the intake reconcile sweep already does — so there's no external cron to set up, and, the part that matters most, it never runs on the read path, so a question is never slowed by it;
- it finds suspects cheaply first, by vector nearness — the pairs of types sitting suspiciously close in meaning-space are only *candidates*; nothing is ever merged on distance alone, the same rule the router already lives by;
- it asks the model, one pair at a time, the plain question distance can't answer — *are these two genuinely the same kind of thing?* — the discipline that tells a sprint from a marathon;
- it stitches the confirmed pairs into groups: if it confirms A≡B and B≡C, that isn't two separate merges, it's one family of three, resolved together;
- it lets the model pick which name in each group survives as the canonical one;
- and it collapses the rest into that survivor — re-pointing every fact link and every child type from the losers onto it (carefully, so a fact already linked to both doesn't collide), leaving each loser behind as a **redirect** rather than deleting it so any stray reference still resolves, and dropping the loser's vector so recall stops offering it.

That last choice — redirect, not delete — is the event-log principle once more in miniature: nothing that was ever true is erased; we only resolve what's *live*.

## the two clocks, laid

The migration went in first, and it is small on purpose: one column on the fact table and nothing more. `created_at` was already there carrying the telling, so the only thing to add is `happened_at`, and it is nullable — that nullability is the whole design: a fact that named no moment stores a real SQL null, not a guessed time, and a reader stands `created_at` in for it. The collapse itself — and any index over the effective time it sorts on — deliberately did *not* land here; that is the read path's to build, and the write path's job is only to fill the column honestly. The heading still says *two clocks* because that is what a fact now carries — `created_at` for the telling, `happened_at` for the event — even though only one of them was new.

## the one particular promoted: reading time off the fact

Then the write path learned to read time out of a fact. It is one model call, given the raw line and the reference instant the fact was told at, and it either returns a resolved timestamp or null — the single particular the otherwise-thin path promotes into structure, because nearly every question a diary is ever asked is a question about time. It reads the fact whole, once, like the naming step, since a fact happens at one moment whatever kinds it touches.

This is where reading it against the live model earned its keep, twice over. The first pass looked right on the easy cases — "yesterday" resolved to the day before the reference, an absolute date passed straight through — but on "I live in Strasbourg," a fact with no *when* at all, the model handed back the reference instant itself rather than null. That is exactly the failure the nullable column exists to prevent: it would have fabricated that the statement "happened" at the moment it was typed, and quietly filled the column that was meant to stay empty. The cause was the prompt treating the reference as available for use rather than strictly as the anchor for relative cues; the fix was to say so in as many words — the reference is only for turning "yesterday" into a date, it is never a default, a fact with no time expression returns null. Read live again, it does: "I live in Strasbourg" and "my favourite colour is blue" come back null, while "last Tuesday," "this morning," and a bare date all resolve to the right instant.

## the merge pass, built and watched work

The garbage collector came in as its own module beside the router, the shape the plan drew: a cheap all-pairs vector scan nominates the near-twins (exact, not approximate — on a store the size of one life's concepts that is both simpler and right), the model confirms each pair one at a time, union-find stitches the confirmed pairs into families, the model names a survivor per family, and each collapse re-points every fact link and child type onto that survivor, tombstones the losers as redirects, and drops their vectors. It runs in-process on a slow daily cadence, the same way the intake reconcile sweep already runs — so there is no cron to set up on the box, and it never sits on the read path. The fact-link re-point is idempotent, so a fact already filed under both a loser and the survivor collapses to a single link rather than colliding with the join table's key.

Watched against the live models, the mechanical round trip holds exactly as drawn: the cheap pre-filter nominated the one near-twin pair — "a session of physical exercise or training" and "a bout of physical training or working out," 0.116 apart — and left the unrelated `phone_call` (0.31 and further) out of the model's sight entirely, and the faked-model suite proves that where a merge *does* fire the collapse re-points every fact link onto the survivor, tombstones the loser as a redirect, drops its vector, and leaves nothing dangling, all in one transaction. The plumbing is right. But on that live pair the pass merged *nothing*: asked with thinking off whether the two definitions name the same kind of thing, the small model said no. That is not a fault in the pipes — it is the deliberate limit the next section is about, and it is why the smoke rolls itself back having proven the round trip without having anything to actually merge.

## a model that can reason or obey, but not both — and here, obey wins

The merge pass surfaced the sharpest lesson of the session. The router's judgments all run with *thinking off* — they are fast classification calls whose reply is held to shape by a decode-time grammar compiled from the reply's schema, so the model can only emit conforming tokens. That works because those calls don't need to reason. The merge confirmation is the one judgment that genuinely *would* profit from reasoning: asked with thinking off whether "a session of physical exercise or training" and "a bout of physical training or working out" are the same kind of thing, the small model says *no* — flatly, repeatably; asked the same question with thinking on, it reasons through the synonymy and says *yes*.

For a moment that looked like a reason to let the offline calls think. But thinking and the grammar are mutually exclusive on this model: apply the schema as a decode-time constraint and the model emits its JSON immediately and never reasons at all — worse, the answer lands in the response's *thinking* channel and the constrained channel comes back empty. It can be forced to a shape, or it can think, not both. So "let the offline calls think" is not free: it means carrying a *second* call mode — thinking on with the grammar dropped, the reply validated only after the fact — alongside the fast one.

And the box this runs on has a harder requirement than the merge pass's recall: it has to answer quickly on the local hardware I actually have. A reasoning trace is latency and tokens spent, and even off the read path it is a second code path to carry for a gain that only ever helps the *subtler* synonyms. So the call went the other way, deliberately: thinking is off everywhere, and not offered. Every judgment, write-path or offline, is one fast grammar-constrained call through the same boundary. The cost is named plainly rather than hidden — the merge pass confirms only the clear-cut twins and leaves the subtler synonyms standing, so it *under*-merges. That is the safe direction: it never wrongly fuses two distinct kinds, it only fails to fuse two that are genuinely one, and a leftover duplicate is a smaller harm than a wrong merge. It is the honest floor a fast local model works from. The reasoning path is not gone, only off the table — it waits for a machine that can afford it, and this is not the moment to pay for it.

## the one knob left unpinned: how near counts as a twin

The merge pass leans on a single number the session set but did not truly *settle*: the distance below which two types are even considered possible twins. It is worth being clear about what that number is and isn't, because it is the pass's one real tuning knob. It is a cheap pre-filter, not the verdict — the model still makes the same-or-not call — but it decides which pairs the model ever gets to see, so a badly-set line quietly bounds everything downstream of it.

What it is: an *absolute* distance in vector space, where zero means two types mean the same thing and larger means more different — the same kind of fixed measurement as a length in metres. It is deliberately not a percentile or a rank; nothing about it is "the closest N% of pairs." The default is 0.2, and it is hand-set, not derived from any distribution. The live smoke gave it its only evidence so far: the two genuine synonyms sat 0.116 apart, the unrelated pair 0.31 and beyond, so 0.2 fell cleanly between them and separated the real twin from the stranger. But that is a single toy run of three invented types — it says 0.2 worked *there*, not that it is right for a real diary, where true synonyms may well sit further apart than this one example did.

The knob has a tradeoff with a sharp edge on each side. Set it too low and real duplicates never clear the bar, so the model is never asked about them and they are *silently* never merged — the worst kind of failure, invisible. Set it too high and the model is handed piles of plainly-unrelated pairs to rule on, burning calls for nothing. The tempting escape is to stop using a fixed number at all and set the line from the data instead — "the closest few percent of all pairs are candidates" — which would self-adjust as the vocabulary grows. That is a real option, held in reserve, but it carries a catch a fixed cutoff avoids: a percentile *always* selects something, so a store that happens to hold no duplicates this week would still shove a batch of false matches at the model, whereas an absolute cutoff can correctly select *nothing* when the nearest two types are genuinely far apart. So the fixed number stays for now — and the smoke prints the real pairwise distances on every run precisely so the line can be moved to wherever a real store's true duplicates actually cluster, rather than guessed. A percentile approach, if it ever comes, would need a floor under it for exactly the empty-store case.

## where this leaves the write path, and what's still ahead

The write path now files a fact whole: its two clocks fixed, its event time read off the line or left honestly null, its concepts named and each routed to a type, and the thin payload persisted — while an offline sweep keeps the vocabulary from silting up with the duplicates that routing by similarity is bound to breed. All of it is proven twice: the automated suite exercises every seam with the models faked at the network boundary, and two by-hand smokes drive the real `nomic-embed-text` and `qwen3.5:4b` against the real store and roll back, so the live behaviour is watched, not assumed.

What is still ahead is the half this whole session was arguing *for*: the read path. Everything laid here — the null that collapses to `created_at`, the timeline the two clocks make orderable, the clean deduplicated vocabulary the merge pass protects — is latent until something reads it back out to answer the symbiot. That is the next stretch, and it is where the fixed retrieval budget, the fast-then-deeper answer, and the type-navigated recall this session kept talking about stop being motivation and become the thing being built.

## the read path opens: the fast answer, and the first reply that is really the diary's

Everything the write path laid is latent until something reads it back to answer, and this is where that begins. It begins, deliberately, with the half of the read path the symbiot actually waits on — so it is the half that has to be quick. When the human says something, an answer should land without a pause, built from what the diary already holds; the deeper reading-by-meaning comes after, off to the side, as a second pass that is not this brick. So the first brick is the *fast* answer: a plain text reach into the raw facts, and the first real reply composed from what it finds.

### the librarian: a fast lexical reach into the raw facts

The fast reach is lexical, not semantic — and lexical does not mean crude. It leans on everything Postgres full-text search offers: every fact's raw text turned into a searchable form and ranked for how well it answers the words the symbiot just used, with fuzzy trigram matching alongside it so a near-miss — a typo, a half-remembered word — still surfaces the fact it was reaching for. No embeddings, no walk over the vocabulary; that meaning-based reach is the slower second pass, and it is not here. This one searches the facts' own words, and *only* their words — the raw line as it arrived, the durable truth the whole store is built on — for two reasons: that search can answer before the vocabulary is even populated, and it is quick enough to sit on the path the human is waiting on. It reads, and never writes; the store is the write path's, and this brick only queries it.

### two languages, because the emotive entries are in French

A question surfaced while building the reach that the write path never had to face: what language is the diary in? The honest answer is two. The symbiot lives mostly in English but drops into French for the emotive entries — the ones written angry, raw, unguarded — which are exactly the ones a reply can least afford to miss. And full-text search is language-specific: an English analyser folds English inflections ("running" and "ran" become one) and strips English filler words, but turned on French it folds nothing and strips none of the French filler, so "énervé" and "énervement" sit as strangers and the very entries that matter most would retrieve worst. Reading French through an English lens quietly makes the French half of a life second-class, and that was not acceptable once it was named.

So raw_text is indexed under both analysers, English and French, and a fact is a hit if either one matches — each language's entries folding and ranking on their own proper rules rather than one read through the other's, and no per-entry language tag to maintain. It is honest about the limit: the French stemmer folds many forms but not all — it folds "énervé" and "énervement" to one stem, yet leaves the past participle "couru" where it is, out of reach of "courir" — and it is exact-word matching plus the language-agnostic trigram pass that catch much of what the stemmers let slip. This is not full bilingual *understanding* — that is the deep semantic pass's job, later — but it is the difference between the French half of the diary being searchable on its own terms and being an afterthought.

### the effective time the write path left for here

The two clocks come due at this point. A fact carries `happened_at` when it named a moment and null when it didn't, and the write path deliberately stopped at filling the column honestly — it left the standing-in for later, for exactly this reader. So this is where the null collapses: a fact is ordered on its *effective* time, when it happened if it said so and otherwise when it was told, and this brick owns both that collapse and the index that keeps sorting on it cheap — neither of which the write path built.

### the first reply that is really an answer

Until now the kernel has answered every message with a fixed line — one for a recognized symbiot, another for a stranger — a stand-in holding the seam open for the intelligence meant to grow there. This brick grows it. The context the librarian gathers is handed, together with the symbiot's message and the machine symbiot's own persona, to the model, which composes the reply in its own voice — the first answer the kernel gives that is read off the diary rather than canned.

This asks something of the model boundary it hasn't had to do yet. Every call so far returns strict JSON held to shape by a decode-time grammar — a score, a verdict, a coined name. A spoken reply is none of those: it is free prose held to no schema, so it needs a second way through the boundary — a generate-text path beside the generate-JSON one, with nothing constraining the tokens but the prompt. And because this is the answer the human waits on, it takes the same hard local-speed stance the rest of the session settled: the fast reply is composed with thinking off, quick over deliberate; the deliberate reading is what the later second pass is for.

### the budget the boundary keeps: fitting a prompt to what the model reads well

Folding memories into a reply raises a question the write path never had to ask: what happens when there are too many of them? A prompt is not free — the model has a ceiling on how much it can read at once, and, less obviously, a much lower ceiling on how much it can read *well*. Long-context benchmarking has settled this into a rule of thumb: a model reliably uses only half to two-thirds of the window it advertises, and past that its recall quietly frays — facts in the middle get lost, details get invented — with no error to show for it. So qwen3.5:4b's window, generous on paper at around 256K tokens, has an honest working figure nearer 128K, and it is that working figure — the size it reads *well*, not the size it will accept — that the kernel should hold itself to.

This is why the kernel now keeps a small map of the models it uses — each one's provider, its name, and its *optimal* context window, recorded deliberately as the size it reads well rather than its maximum — and consults it before every generative call, free-text reply or JSON judgement alike. The assembled prompt is measured against that optimal, and the measuring is done locally, with the industry-standard token counter (tiktoken), kept conservative by a margin because it only approximates qwen's own tokeniser rather than matching it. Under the figure, the call goes through untouched. Over it, the boundary summarises the prompt down to fit — but only its *context*, the folded-in memories, never the instructions wrapped around them, because summarising the whole prompt could quietly delete the very lines that say answer as yourself, or return JSON only.

In practice, with a reply prompt a fraction of a percent of that 128K figure, this guard will almost never fire. It is laid now anyway, as a backstop rather than a daily worker — the brick put down before the first prompt that would have overrun it, not after — so that the day a diary grows dense enough, or a future caller careless enough, to push against the ceiling, the boundary already holds the line rather than handing the model a prompt it will read badly and answer worse.

### what this brick is not

It is only the fast half. The reach by meaning — the embeddings and the vocabulary walk — the second, richer reply that follows a beat later, and the judgement of whether that deeper pass adds enough to be worth sending at all: all of that is the next brick, not this one. And one seam stays open on purpose: live ingestion isn't yet wired into the message flow, so a fact is filed only by hand today, not as messages arrive. Until that is wired, the librarian reaches into a shelf that live use leaves empty, and the real reply composes over empty context — correct, and proven against a seeded store, but not yet fed by the live path. Closing that seam is its own brick.

## feeding the diary: live ingestion, exactly once

The read path can answer off the diary now, but in live use the diary stayed empty — facts were filed only by a by-hand script, never by the kernel as messages actually arrived — so the librarian reached into an empty shelf every time. This brick closes that: every message the symbiot sends is filed into the diary through the write path already built over the last two sessions (name the concepts, read a time off the line, route each to a type, persist the thin payload), so the memory the read path leans on fills itself as the conversation happens, rather than waiting on a human to run a script.

The tempting place to put that is in the worker, right after the reply — and it is the wrong place. Filing a fact is slow: it is a handful of model calls (the naming, a temporal read, a recall-and-re-rank and maybe a mint per concept, then the embeddings) where the reply is a single one. Hanging that off the reply would make the symbiot wait on bookkeeping they never asked to watch, breaking the one promise the fast path makes. So ingestion is decoupled entirely — the answer is sent back the instant it is ready, and the filing runs in parallel, after and apart: a background sweep in its own thread, on its own cadence, beside the reply workers and the reconcile and garbage-collection sweeps already running, filing the messages that are ready and never sitting on the path the human waits on. There is a quiet benefit in that separation too: the message being answered is not yet in the diary when its own reply is composed, so a message never retrieves itself as its own context — the reply draws on what came before, and the message joins the record a beat later.

Every message is filed, and filed *exactly* once, and that "exactly" earns its weight in a diary: a silently dropped entry is a hole in a life, and a duplicate is noise that spends the retrieval window saying the same thing twice. Rather than trust timing, or a flag a crash could leave half-set, the guarantee is pinned in the database — a filed fact carries the id of the message it was made from, under a uniqueness constraint, so the same message can never be filed twice however the sweep is interrupted or retried. The sweep's eligibility is the exact mirror of that: an authed message that has reached a terminal outcome and has no fact yet bearing its id. So a message filed is a message excluded from the next pass, and a message whose filing crashed before it committed is still eligible and simply picked up again — no drop, no double, enforced by the schema rather than by the sweep having to be careful.

Two lines are drawn deliberately. Only the symbiot's own messages are filed — an anonymous visitor's line is never written into the symbiot's diary, the same line the reply already refuses to cross. And every message is filed as it came, questions and passing remarks alongside the journal entries, because the intake record already *is* the diary of record and drawing the line between "a fact worth keeping" and "a passing question" is a judgement — another model call, another thing to get wrong — better added when the noise proves it is needed than guessed at now. What is here is the plumbing that makes the diary fill itself, honestly and without loss; refining *what* is worth filing is a brick for later.

## the path run by hand, from the shell the symbiot uses

Everything above was proven twice at the seams — the faked-model suite, and the by-hand smokes against the real models — but not yet driven the way the symbiot will actually drive it: through the shell, in a browser, against a live kernel. This session closes on that run. Both halves up, the store a clean slate, every row watched from the database as it appeared.

The security seam held first, because it is the one that matters most. An unauthenticated line, typed at the prompt with no session, came back with the bare stand-in and nothing more: the row carried no symbiot, its answer was the literal stand-in string rather than a composed one, and the diary — the facts, the fact-to-type links, the vocabulary itself — never moved. No model was spent on a stranger, and the ingestion sweep's eligibility gate kept that anonymous line permanently out of the diary, exactly as the write path promised. An unauthenticated caller cannot draw a reply out of the model through The Joy, and now that is watched from the outside rather than argued only from the code.

Then the authed path, against the empty shelf the last brick warned about. A recognized symbiot's message — deliberately one inviting a fabricated memory, asked of a diary holding nothing — composed a real reply in the machine's own voice that declared a clean zero rather than inventing a past it did not have. The message then filed itself into the diary on its own, through the background sweep, exactly once, coining from that single line its first two types from use rather than from any seeded schema. The read path answered, and the write path fed it, both live and both unattended.

One real defect surfaced in the running of it, and was fixed before the commit. A composed reply carries paragraph breaks, and the shell was writing those breaks to the terminal raw, where the terminal needs a carriage return alongside each line feed — so a multi-paragraph answer staircased rightward across the screen instead of standing in a clean column. The fix converts every embedded break as it is written, so continuation lines land flush-left; the reply now reads the way it was composed.

One question was raised and deliberately set aside: streaming the reply into the shell as it is generated, rather than landing it whole. It is not a small change — the reply is composed inside a killable child process, stored atomically, and delivered exactly-once by a poll the shell already runs, and a token stream cuts against all three — so it is logged as its own piece of work, to pick up once the read path has fully settled, not folded into this commit.

## what running it live made plain

Two needs came into focus in the running of it — not as surprises, but as the next rungs a working loop makes concrete, and both are cause for confidence rather than concern: they are the problems you only get to have once the thing works end to end.

The first is that the persona wants to be tweakable from inside the conversation. The machine symbiot answers in a voice spliced from two files, one open and one private, and today that private half is edited only by hand, on the box. Once the human symbiot is actually talking to it, the natural next move is to reshape that private voice in the conversation itself — terser here, drop that there — and have the change hold. The read path proved the voice lands; the next brick lets the human symbiot tune it without leaving the shell.

The second is that the model behind the loop needs to be both smarter and faster. What runs today is a small Qwen served on remote CPU — right enough to prove every seam, but dim and slow in a way that shows the moment you actually talk to it. This was always the expected shape: settle the loop against a cheap local model first, then put a real engine behind it. The next brick moves the generative calls onto a bigger Qwen on GPU-backed hosting, with a fallback provider for when the primary is down, so the same proven loop answers with more sense and less waiting.

What ties both to a good feeling rather than a worried one is the thing underneath them: the whole conversational loop — a message in, the right memories retrieved, a reply composed in the machine's own voice, the message filed back into the diary — runs end to end, live, from the shell the symbiot will actually use. Neither of these is a repair for something broken. They are the refinements a working thing earns.

## onto the client Ollama itself advertises

A smaller correction landed alongside the run. The kernel had been talking to Ollama by hand — a raw POST to `/api/generate` for the generative calls and `/api/embed` for the embedder — rather than through the Python client Ollama publishes and documents. Both call sites now go through that client. The gain is not a new capability but ownership of the contract: the shape of the request and the reply is the maintained library's to keep current as Ollama's API moves, not ours to quietly drift against. The client is built fresh on each call, so its connection pool is never shared into the killable child the reply is composed in, and every setting the kernel fixes — thinking off, streaming off, the decode-time grammar that holds JSON to shape, the temperature nailed to zero for the router's judgments — passes straight through unchanged. The faked-model suite moved onto the same seam and still holds green, and the three by-hand smokes were re-run against the real models, the whole write and read path proven live through the new boundary.

## a real engine behind the loop: Scaleway, with fallbacks home

The second of those two needs is now closed. Every generative call the kernel makes — the spoken reply, and the whole router beneath it (re-rank, the grey-zone gate, the garbage collector's same-kind and survivor judgments, the concept and time extraction) — moves off the small Qwen on remote CPU and onto GLM-5.2 served on Scaleway's GPU-backed inference. Moving the router too, not just the reply, was the deliberate call: the dim replies and the dim ontology were never two problems, only one underpowered engine showing in two places. The proof came in the running — where the old model had merged nothing on the duplicate-collection smoke, the new one reads two differently-named training types as the same kind and collapses them; where the old reply invented percentages and metric names it had no source for, the new one states plainly what the diary does and does not hold. Embedding is the one call that stays on the box: the vector tables are typed to its model's width, so it cannot move without re-embedding the whole store, and it earns nothing by moving.

Sending the symbiot's words to a provider you do not own means planning for the provider being down, so generation is a ladder rather than a single call: Scaleway first, then Mistral's own web API, then — only if both clouds are unreachable — the local Qwen that used to serve every call, kept wired as the last resort so the loop never fully stops. The check is a plain per-request failover, not a circuit breaker holding state between calls. That is a considered choice, not a shortcut: the reply is composed inside a killable child process, so any memory of "Scaleway is down" set in that child dies with it and never reaches the next call — a breaker could only ever help the background router calls, at real added complexity, to save paying one timeout during an outage that, for a single symbiot, barely registers. What the ladder does insist on is the difference between an outage and a mistake. A transport error, a timeout, a 5xx, a rate-limit — those are the provider failing, and they fall through to the next tier. A 4xx is the kernel's own bad request or a bad key, and it surfaces at once rather than falling through, because the next provider would fail on it identically and the fall-through would only bury the bug.

Each provider is reached through the client it actually publishes, learned from its own documentation rather than guessed. Scaleway speaks the OpenAI wire protocol, so it is reached through the OpenAI Python client pointed at Scaleway's endpoint, with the SDK's own retries turned off so an outage fails fast to the fallback instead of the library quietly retrying first. Mistral is reached through its official client, at Mistral's own API — pointedly not the Mistral that Scaleway also hosts, since the entire reason for the fallback is Scaleway being unreachable. Two provider specifics only the docs settle, and both bit before they were read. Turning GLM's reasoning off is done with Scaleway's documented control, a `reasoning_effort` set to none — not the field the model's own card advertises, which Scaleway's gateway explicitly does not honor and silently ignores. And holding the router's output to an exact shape is done through each client's blessed structured-output call, the one that hands the decoder a strict schema and binds it: a plainer request that merely names the schema turned out to be only a hint on Scaleway, and the model answered straight past it, returning a bare word where a JSON object was asked for. The kernel's own validation still stands behind the boundary as the guarantee, whichever tier answered, exactly as before.

One truth came only from asking the live service rather than the model card: the identifier Scaleway answers to is the plain lowercase name, not the org-prefixed one the card prints. The catalog the endpoint itself lists is the authority there, and the wrong guess was a clean 422 that the ladder correctly refused to paper over — the very 4xx-surfaces rule, earning its place on the first real call.

It is proven at every seam. The faked-model suite moved onto the new client boundary and holds green, now with cases that force Scaleway down and watch Mistral answer, force both clouds down and watch the local tier answer, and force a 4xx and watch it raise without any fallback being tried. All three by-hand smokes were re-run against the real GLM-5.2 — ingestion, duplicate collection, and the composed reply — and a deliberate outage, the primary's address pointed at a dead host, confirmed Mistral picks up both a free-text reply and a schema-bound one for real. The loop that was proven end to end last is the same loop; it simply answers now with more sense and less waiting, and it keeps answering when a provider blinks.

## two settings the ladder still passed through on defaults

The engine was in place, but two call settings flowed through the boundary untouched, each left at whatever the provider defaulted to — and a default is not a decision. The first is how long a single reply may run. None of the three tiers had been told, so each fell to its own: GLM to sixty-odd thousand tokens, the local model to no cap at all, generate-until-the-window-fills. That is a runaway waiting for a bad prompt — a model that loops or repeats could spend the whole latency budget before anything stopped it. So every generative call now carries an output ceiling. It is deliberately not the input window's twin. The window's optimal sits *below* the advertised maximum for a measured reason — recall frays in the upper half of the window — but an output cap has no such curve behind it: a reply is not worse for being allowed to run longer. So the question it answers is the opposite one. Not "how far below the maximum must we sit to keep the quality" but "how high can we safely go, so a genuinely long reply is never cut short, and no higher." The honest way to find that ceiling was to stop reading spec sheets and ask the live endpoints. The model cards mislead — GLM's says it can emit a hundred and thirty thousand tokens — but the provider is the real authority: Scaleway hard-caps GLM's output at sixteen thousand three hundred and eighty-four, sized by its own rule that a response return inside five minutes, and a request over that comes back a plain four-hundred, the kind of error that does not fall through to the next tier but surfaces and kills the call. So that verified figure, not the card's, is the true ceiling. Mistral, probed the same way, enforces no request-time cap at all — it accepts an absurd figure and simply stops when it is done — and the local model is bounded only through Ollama's own knob. Sixteen thousand three hundred and eighty-four is the highest the most-constrained tier will honour, and it is held across all three, so a fallback never truncates a reply shorter than the primary would have given — one ceiling, at the top of what is safe, the same reason one context window fits them all. It is read per tier, from the model about to answer rather than resolved once from the primary and carried down, so each is held to a cap it actually supports. The summariser asks for more when it needs it — condensing an over-budget prompt, it can legitimately want room up to its whole target — but that request is now clamped to the answering tier's ceiling rather than granted outright, because a target sized to the context budget can run clean past what Scaleway allows, and asking for more than the cap is that same four-hundred, not a longer summary.

The second was sampling. The router's judgments had always been pinned to zero, so the same inputs score the same way twice; the spoken reply had been left at each provider's default warmth, on the thought that prose reads better a little warm. But the reply can be composed by any tier of the ladder, and each provider warms its own default differently, so the same diary could come back in a different register depending on which of the three answered. Pinning the reply to zero as well — and the summariser with it — closes that. It does not make the three models speak as one voice: they are different models and read differently, and zero cannot erase that. What it removes is the avoidable part, the sampling randomness stacked on top of the differences between the models, so each tier now answers as steadily as it can and the same diary tends to reproduce the same reply. The faked-model suite moved onto both settings and holds green — holding the reply to its ceiling, watching that ceiling resolve to each tier's own figure as the ladder falls through, and watching the summariser's oversized target get clamped back to what the tier accepts.
