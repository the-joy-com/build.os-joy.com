# session 23 of building The Joy in the open

## where session 22 left things

Last session was two threads braided together. The first crossed a line the loop had been circling: it learned to *act*. The tool-calling machinery went in — a real registry born correct with a single inhabitant, the one-shot reminder — with the discipline that the model only ever emits a *name* and code alone runs the effect, and the reach fanning out through a notification layer rather than leaning on one push. The reminder earned a clock to stand on first: the symbiot got a timezone of its own, named in plain words and resolved through the structured-output boundary, so "remind me at nine" finally knew whose nine.

The second thread was time, and it stayed unfinished on purpose. A handful of honest, bounded fixes shipped — the stored instant and the stated "now" brought onto one calendar, the recent turns carrying the clock they were said on, the deep follow-up given a present to reason in, and the diary facts rendered oldest-first instead of relevance-first so a ranked list stops reading as a timeline. But underneath them sits the real rot, and I named it rather than pretended the render tweaks cured it: the fold flattens a week into one timeless paragraph, there is no notion of *supersession* so a stale fact sits coequal with the live one, and there is no identity model for people at all — so one person splits into two and two people fuse into one. That is logged as a single urgent bug owed a dedicated design pass, not something this stretch closed.

## what this session is about: reviewing what's not yet committed

Everything from the back half of that work is sitting uncommitted in both repos, and this session opens by reading it back critically before it lands. Two bodies of change.

In the **kernel**, the notification layer is the new arrival. `services/loop/notify.py` is the single dispatcher — one uniform payload (title, body, a pointer back to the durable inbox record that stays the source of truth) fanned across every channel the symbiot has, with the reminder narrowing to the channels named when it was set and an unprompted missive fanning everywhere. Two silent filters sit under the fan-out: a channel that doesn't exist is dropped, and one the symbiot has globally switched off is never fired. That standing switch is `services/memory/notification_prefs.py` — a store that holds only the exceptions, defaulting every channel on — driven by a new `/notifications` route in `main.py`. Two migrations carry it: `0018` gives the reminder row its per-fire `channels` array, `0019` stands up the `notification_preference` table. Around that, the changes the layer pulled in its wake: `missive.py`, `enrichment.py`, and `reminder.py` now all reach out through `notify.dispatch` rather than each knowing how to touch a browser or a mailbox; `push.py` grew the real TTL and high urgency so a nudge survives a sleeping phone; and `zone.py` gained `local_date` and the shared local-now line that the reply and deep-follow-up render seams both read. The rest of the diff is the time-things and their tests — `conversation.py` carrying `created_at` through to the turn render, `reply.py` and `enrichment.py` stamping clocks and ordering facts oldest-first.

In the **shell**, the smaller half: `src/notifications.ts` is the new `/notifications` flow — read the current per-channel state, offer it as a checklist to tick and untick, tell the kernel each change, print what moved — cut to the same authed-only, hidden-until-session pattern the timezone command set. `commands.ts` and `kernel.ts` wire it in. `zone.ts` gained the read-before-write so `/timezone` opens on the value it already holds instead of prompting blind, and `notify.ts` closed the login gap where a browser that subscribed while logged out was never adopted under the symbiot's name.

That is the code to walk through. The point of the read is the usual one — that the invariants the narrative claims are actually the invariants the code holds — before any of it is committed.

## a requirement the first onboarding surfaced: a fully local symbiot

Yesterday I tried to bring a friend onto the project, and the attempt failed in a way worth logging rather than patching around, because it named a shape the build had quietly assumed and never questioned. The whole kernel is wired to three paid or hosted dependencies that I happen to have and he happens not to: the generative and embedding calls go out to Scaleway/Mistral, which costs money he can't spend; first contact and the login code ride Gmail, which he hasn't set up and shouldn't have to; and the model map names remote models as though those were the only kind. What he *does* have is the one thing I'd never designed for — a capable server at home already running decent models under Ollama. So the machine that was built to be private-by-instinct turned out to have a hosted spine, and a person who wanted the most local version of it couldn't run it at all.

The requirement, then, is a **local-only mode**: a way to stand the symbiot up with no paid API and no Gmail, leaning entirely on models the operator already runs. Two seams have to give for that to be true.

The first is **authentication without email**. Today the one-time code is delivered by Gmail and nothing else, so "no mail wiring" means "no way to log in." The requirement is to deliver the OTP by some other channel when mail isn't configured — the specific channel is a design question for the plan, not something I'm fixing here — so that identity, which the whole authed surface stands on, doesn't depend on a mailbox the operator never set up. This is the same provider-independence instinct the notification layer already carries, owed now to first contact itself.

The second is **operator-chosen models, wired through a command rather than baked into config**. The kernel leans on more than one model for more than one job — a heavier one composes the reply, a lighter one reranks, and the embedding model is its own thing again — and today those are named in config and assumed remote. The requirement is a dedicated, authed command by which the operator registers the models they want to use and the characteristics each carries (its endpoint, its context window, whatever the kernel needs to drive it well), with **sensible defaults filled in** where they don't say — an optimal context window and the like — so a bare Ollama model name is enough to get running and the fine print is optional. The command has to let them **edit and delete** registered models too, not just add: a model per role (reply, rerank, embedding, …), each swappable without touching config or code. This is the model map turned from a compile-time constant into a piece of the symbiot's own configurable state, the way the timezone and the notification switches already are.

Logged as the requirement only. The design — how the OTP travels when mail is off, exactly how a locally-registered model threads through the adapters that today assume the remote ones, how the roles are keyed and defaulted — comes as its own pass, to be validated before any of it is built.

## the direction that pass took: two tables, fully dynamic, seeded from what was hardcoded

Talking the design through settled it into a shape sharper than the requirement guessed at, and the sharpening came from pulling apart a word I had been using for two things. "The model config" was hiding two separate jobs. One is the *catalog* — the set of models the kernel knows how to talk to, each with the characteristics it must be driven by: which provider serves it, the window it reads well, the output ceiling it's held to. The other is the *assignment* — which model, out of that catalog, does which job: one composes the reply, another reranks, another gates the enrichment follow-up, and so on down the roles. Today the first lives as a hardcoded dict in `adapters/models.py` and the second as a scatter of config constants (`REPLY_MODEL`, `RERANK_MODEL`, and the rest defaulting off them). The requirement asked for both to become editable; the clean way to grant that is to stop conflating them.

So the direction is **two tables, both fully dynamic, both seeded from exactly what is hardcoded today**. A catalog table, one row per model — name, provider, optimal-context, output cap — seeded with the four the kernel already ships (`glm-5.2`, `mistral-large-latest`, `qwen3.5:4b`, `nomic-embed-text`). And an assignment table, one row per role pointing at a catalog entry — seeded from the current config defaults, so the running symbiot's behaviour is identical on the first boot after the migration as it was before it. Nothing changes for me; everything becomes reachable for my friend. The seed is the safety net: the dynamic store starts life holding precisely the values that were compile-time constants, so "make it configurable" costs no behavioural drift on day one.

Both tables are **global, box-level** — not keyed per symbiot the way the timezone and the notification switches are. That is the one place this diverges from those commands' pattern, and deliberately: a model is a property of the machine and the Ollama it can reach, not of a person's perception of their own day. A box runs the models a box runs. The catalog seeds its built-in rows the way the tool catalog already reconciles itself from code at boot — the code stays the source of truth for the researched, verified specs (Scaleway's real output cap, the optimal-not-advertised windows), pushed into the table non-destructively so an operator's own added rows are never trampled. And the operator reaches all of it through a new authed `/models` command cut to the same cloth as `/timezone` and the newly-checklisted `/notifications` — list what's there, add or edit their own models, delete them, and point a role at whichever model they please, e.g. swing `reply` onto a local `qwen` to run the whole loop on their own box.

The one consequence I want named rather than buried: once the tables exist, `adapters/models.py`'s dict stops being *read* at runtime — it becomes the seed definition and nothing more, and every place that today looks a model up in it (the budget guard, the output-cap resolver, the two direct `optimal_context` reads in `conversation.py`) is rewired to resolve against the store instead. That resolver is the widest, most invasive part of the work, and it is the part that has to stay green through the whole rewiring, because it sits under every generative call the loop makes.

Two scoping lines held. `nomic-embed-text` stays a **hard requirement**, not an editable choice — every box can run it, and the embedding model is tangled in the pgvector column dimensions and the per-model vector tables in a way that makes swapping it the heavy re-embed migration from last session, not a command edit. And the OTP, when mail is off, travels the simplest honest local channel there is: the code is written to a gitignored `OTP.txt` at the kernel repo root, with a console log line signposting when and where, so an operator who controls the box — the same trust root as the SSH key — reads their own code without a mailbox ever entering the picture.

## and then built, across the spawn boundary the design hadn't reckoned with

Building it went as designed on the login half and turned up one real surprise on the model half. The login half is small and clean: the kernel already sent its code through an `EmailClient` interface, so mailboxless delivery is just a second implementor of it — a `FileEmailClient` that writes the code to `OTP.txt` and logs where — and which one a box gets is a single config choice made at startup (Gmail wired → email it, no mailbox → write the file). Nothing above the interface knows or cares that delivery went to disk. Enumeration-safety came for free: a local file has no external observer at all, so the login reply stays as silent as it ever was about who's registered. A gitignore line, a note in `.env.example`, and two tests that pin the write and the overwrite-don't-append, and that seam was done.

The model half is where building taught me something the design had glossed. I had said the resolver — the thing that turns "the reply role" into a concrete model and its characteristics — would be the widest change, and it was; what I hadn't reckoned with is *where* the reply is actually composed. It runs inside a killable child, and that child is a **spawned** process, not a forked one — a fresh interpreter that inherits none of the parent's memory. So a cache the parent loads from the store is simply not there in the child that does the composing. The parent already resolves everything the child needs and passes it across the spawn boundary — the diary facts, the conversation, the local clock — so the honest fix in that spirit would have been to resolve the model in the parent and pass it down too. But that meant threading a new value through every child's argument tuple and every faked signature in the tests, wide and brittle. So I took the other road that the spawn boundary allows: the resolver loads its cache **lazily, once per process**, and a child with a cold cache reads the store itself — a single cheap read of two tiny tables over a direct connection, using the same database URL the parent has. The parent still loads eagerly at startup and never pays that read; the child pays it once, on the box where the database is local anyway. The whole rewiring became a set of one-line swaps at the call sites — `config.REPLY_MODEL` becomes `models.role_name("reply")`, `models.MODELS[…]` becomes `models.spec(…)` — with no change to a single function signature, argument tuple, or existing test. That the suite stayed green through it (296 passing, up from the 256 it sat at a session ago) is the evidence that a resolver sitting under every generative call didn't quietly change what any of them do.

The rest built as drawn. Two tables (migration 0020): `model` the catalog, `model_role` the assignment, both reconciled from code at boot — the builtins upserted so their verified specs track the code, each role seeded from its config default only when unset, so an operator's own choices are never trampled and a fresh box behaves exactly as before. The store enforces its own rules rather than trusting the caller: a builtin can't be edited (a reconcile would overwrite it) or deleted (it ships with the kernel), a model a role still points at can't be deleted (it would orphan the role, and the refusal names the roles holding it), and a role and the model it's assigned must both be real. A bare model name registered by the operator gets sensible defaults filled in — a local provider, a generous window, the builtin output cap — so "point reply at my local qwen" needs a name and nothing else. And the `/models` command in the shell is cut to the same cloth as `/timezone` and `/notifications`: authed-only, hidden until there's a session, opening on the kernel's current state before it changes anything, and looping — assign, add, delete — until the operator is done, re-rendering the kernel's own answer after each change so the screen is never the shell's guess. The one place it diverges from those two is the one the design already named: the config is box-level, not per-symbiot, because a model is a property of the machine, not the person.

What's proven and what isn't, stated plainly. The kernel suite is green and covers the store's rules, the resolver, the route, the file channel, and — the crux — the cold-cache lazy load that stands in for what a spawned child does. The shell typechecks, client and service worker both. What the suite can't reach is the same thing it never could: the live path against a real local model. That a reply actually composes on a box whose `reply` role points at an Ollama `qwen`, that login actually lands a usable code in `OTP.txt` on a box with no Gmail — those are the by-hand smoke, owed before the friend's box is called done. But the machinery beneath them is built, and built so that the fully-local symbiot is now a configuration, not a fork of the code.

## the notifications switch grows a checklist

The first thing the review turned up was in the shell, and it was less a bug than a shape that had settled for less than it should. `/notifications` opened by printing the channels and their on/off state, then asked me to *name* the one I wanted to flip and typed it back — one round trip per channel, and a channel name to spell each time. That's the same "talk at me" pattern the timezone command started as before it learned to open on its current value, and it read as thin the moment I used it: a list of checkboxes is exactly the thing a keyboard should let me walk and tick, not a name I retype.

So the switch grows a real checklist. It opens on the kernel's current answer — the same read-before-write the timezone and notifications commands already lean on, so the screen is the truth before a key is pressed — and renders each channel as a row with a cursor, a `[x]`/`[ ]` box, its label, and the on/off word spelled out beside the box so the state reads even without colour. I move the cursor with the arrows, tick or untick the row under it with space, and settle the whole set with Enter, or abandon it with Esc — one pass over the list, not one round trip per channel. A dim line of microcopy under the rows says exactly that: `↑/↓ move · space toggles · enter saves · esc cancels`.

The seam it opened is where the interesting decision was. The terminal is hand-built — plain DOM, no emulator — and its own doc draws a sharp line: the terminal owns the surface and the *raw key handling*, and the app above owns what a line *means*. A checklist is raw-key handling, so it belongs in the terminal beside `readLine`, not smuggled into the notifications flow reaching into keystrokes it has no business touching. So the terminal grew a `checklist` primitive — a modal read that takes the keyboard for its duration exactly the way `readLine` does, then hands the input line back — and the notifications flow just hands it the channels as data and reads back the final set. The same division of labour that already keeps every other command out of the DOM.

Two things I would not leave soft. First, the input line doesn't just hide while the checklist is up — it drops its focus too, because a hidden-but-still-focused editable would quietly catch the Enter I meant for "save" and commit a blank line behind the picker; the keys are handled in the capture phase so a stray one can't leak past. Second, I would not ship a switch that a phone can't touch. Space-to-toggle is a keyboard gesture, and the shell lives on my phone as much as my laptop, so every row is tappable and there's a `[ save ]` / `[ cancel ]` pair to tap beside it — the keyboard path and the pointer path reaching the same state, neither one the poor relation. And because the kernel still takes one channel per write, the flow walks only the rows that actually changed and reports from the kernel's final answer rather than from what I ticked, so the screen can never claim a flip the kernel didn't take.

The kernel didn't move for any of this — the route already spoke one channel at a time, and that was enough to drive from a list. It's all shell: a new `checklist` on the terminal, the notifications flow rebuilt on it, and a line of CSS so a tappable row shows a pointer. The build is green, client and service worker both; the picker is a browser-and-session thing, so the honest proof of it is driving it live once the review's own changes are committed.

## bringing the shell's front door back in line with the code

While the review had the shell open, I read its README as a stranger would and found the front door lagging well behind the rooms behind it. It still introduced the identity verbs — `/login`, `/logout`, `/status` — as "listed and acknowledged but not wired to a backend," a sentence true weeks ago and false now: they run end-to-end against the kernel's session machinery, code and token and expiry and all. And the whole settings surface that has grown since — `/timezone` for local time, `/notify` for this browser's web push, `/notifications` for the standing per-identity channel switch — went unmentioned, as did any map of what lives in which source file. The individual modules are not the problem; each carries a header comment that states its own contract plainly, and those read well. The problem was the one document a newcomer meets first.

So the effort was the README, not the code comments. I fixed the stale claim, then documented the command surface honestly the way `/help` itself presents it — the screen verbs, the identity verbs, and the personal settings — with the visitor-versus-authed line drawn explicitly, since a caller with no session is shown only what it can use. I gave `/notify` and `/notifications` the paragraph they need to stop being confused for each other: one is a subscription belonging to *this device*, the other a preference belonging to *my identity*, and a channel switched off in the second is dark on every device regardless of the first. And I added a source map — one line per module, an index over the header comments rather than a replacement for them — so a contributor knows where a thing lives before they go looking.

The deeper drift was in the two sections the README already had, and it ran the wrong way round: the capture-loop section still described a page that sends its own lines straight to `/intake` and marks a failure `✗ not delivered`, and it closed by calling an offline queue a "separate capability built on top of this" — a future tense for something that has since been built and now *is* the loop. The reality it had fallen behind: the page never touches the network at all; a typed line is written to a durable IndexedDB outbox first, and the service worker drains it — as one timestamped batch, surviving a full app close, retried on reconnect by Background Sync — so a failed send is `⋯ queued` and held, never dropped. So I rewrote that section to the architecture that actually runs, and added the one it never had a word for: the reply channel. The kernel answers now, and the answer comes back decoupled from the sender — the shell keeps only the message id, never a copy of what it said, and surfaces the reply on its own terms by whichever path reaches it first (the on-page poll a visitor gets for free, a push only sooner, the authed inbox for messages the kernel raised unprompted), each acknowledging only on a real showing so nothing is lost and nothing shows twice. The front door now describes the house that's actually standing.

## a courtesy that knows when to stay quiet: no email while I'm already here

The next thing I want to close is a small rudeness the notification layer commits without meaning to. When the kernel raises a missive on its own, it fans out to every channel there is — the durable `/inbox` record, a web push, and an email — because first contact should never ride on one channel holding up. But that fan-out is blind to a case that's plainly different: the symbiot is *sitting in the shell right now*, watching the tab, and the missive is already going to land in front of them within a beat. In that case the email isn't a second safety net; it's a redundant ping in a real mailbox for something they're about to read anyway. I want the layer to know when to hold the noisier channels back.

The trouble is the kernel has no idea whether anyone is watching. Every route is stateless request-and-reply — there's no live connection, no session presence, no `last_seen` anywhere. "Is the symbiot in the shell right now" is a question the kernel currently cannot answer at all. So the first half of this is giving it the ability to ask.

The material for that already exists, which is what makes this cheap rather than a new subsystem. The shell, while its tab is actually visible, polls `/inbox` every ten seconds — and only while visible; a backgrounded tab goes quiet. That poll is already a near-perfect heartbeat: it fires exactly when someone is looking and stops when they aren't. So presence doesn't need a new channel or a websocket; it needs a timestamp. I'll stamp a `last_seen_at` on the symbiot from that inbox poll, and read "active" as *seen within a tolerant window* — a little over the poll cadence, twenty-five or thirty seconds, so a single missed beat doesn't flip them to absent. One cheap write per open tab every ten seconds, on the box where the database is local anyway.

The second half is what to do with that knowledge, and here the design almost writes itself, because the layer already rests on the right split. A missive is two things stacked: the durable record, which *always* lands and is the source of truth, and the notification, which is *only speed* on top of it — best-effort, skippable, the thing the docstring calls "double up first delivery." When the shell is live and polling, the record already surfaces in-app within that ten-second beat, no push and no email needed. So suppressing the email under presence isn't weakening the guarantee — the guarantee was never the email — it's just refining the principle: don't double up a channel that's already live. The place to do it is where the two silent filters already sit, inside `notify.dispatch`, which drops unknown channels and globally-disabled ones before anything sends. A third filter joins them: when the symbiot is present, drop the out-of-app channels — email for certain, and I'll weigh web push too, since a buzz for something on the screen is the same redundancy.

One line I want to hold, because it's the difference between a courtesy and an override. This suppression belongs only to the *unrequested* fan-out — a missive the kernel raised on its own, fanning everywhere because nothing narrowed it. It must not touch a channel a caller *named on purpose*: a reminder the symbiot explicitly set to email them at nine is an intent, not a courtesy, and "you happened to be in the shell at 8:59" is no reason to silently eat it. So the filter keys on whether the channel was chosen by default or asked for by name — missives narrow under presence; explicit reminder channels never do.

And the tradeoff, named rather than buried, as the last one was. Presence is always a guess — if they close the tab the instant after the last poll, I'll read them as present and hold the email, and they'll lose the *immediacy* of it. But they never lose the *message*: the record still stands, and `/inbox` surfaces it on next open no matter what. Since the whole layer already treats the record as the truth and the notification as disposable speed, that's a loss the design already priced in — a dropped channel costs immediacy, never the message. If it ever proves too blunt I can grade it — present means in-app only, recently-seen means push, long-gone means push and email — but the honest first cut is the single filter, and I'd rather ship that and watch it than build the gradient before I've felt the need.

That's the shape: a `last_seen_at` migration and the stamp on the `/inbox` poll, then the presence filter in `dispatch` scoped to the missive fan-out. The record-always-stands safety is what lets me suppress a channel without flinching, and the visible-only inbox poll is what lets me know when to.

## a landmark in the wall of text: the last line I clicked

A small navigation grievance surfaced while I was living in the shell, and it's the kind that only shows up once the log gets long. The conversation is one unbroken column of monospace — what I sent, what came back, the little markers underneath — and the moment I scroll up to re-read something and then scroll back, I've lost my place. There's no anchor in the wall of text, no "I was here" to scroll toward. Every other surface I use keeps a sense of where my attention last landed; the shell kept none.

So the last line I click now stays highlighted as exactly that — a landmark. Click a message and it takes on a green gutter bar and a faint wash of the same green, bright enough to catch the eye on a scroll-back and quiet enough not to shout over the reply itself; click another and the mark moves there, so there's only ever the one, the place I was last. It's a landmark, not a selection — nothing acts on it, it just holds a spot in the column so my eye has something to return to. The line says it's clickable before I click it, too: the pointer cursor comes up on hover over any real message, and stays away from the blank spacer rows that have nothing to mark, so the affordance never lies about what's a target.

The seam was the same one the checklist taught me to respect: the terminal is hand-built, and its own doc draws the line that the terminal owns the *surface* while the app above owns what a line *means*. A highlight on a rendered line is pure surface — where the eye rests, not what the words say — so it belongs inside the terminal beside the click handling that already lives there, never reaching up into the app. It rode onto the tap the terminal already listens for: a tap anywhere focuses the input so a phone opens its keyboard, and now that same tap, when it lands on a real message line, also marks it. The two compose on one gesture rather than fighting for it. Two guards keep the mark honest — a drag to select text for copying bails before it ever moves (highlighting to copy is not the same act as marking my place), and a blank furniture row carries no text, so it can't become a landmark and leave a stray bar hanging in the gutter. The bar rides in the gutter itself, its left padding cancelled by an equal negative margin, so marking a line never nudges its text sideways by even a pixel.

The kernel didn't move; this is all shell, and small — a landmark the terminal tracks, the mark laid on the existing tap, and a few lines of CSS. Because it's a browser-and-pointer thing the suite can't reach, I proved it the way that surface has to be proven: I drove the built bundle in a headless browser, injected message lines into the live log, and clicked. The mark lands on the clicked line, moves to the next click and leaves the first, never sits on two lines at once, skips a blank row, and the green paints without shifting the text a hair — eight checks, all green against the real build, not a reasoned guess about what the DOM would do.

## and then built: the courtesy filter, keyed on who asked

The presence pass built as drawn, and the one thing it turned up was less a surprise than a line I had to make sure held under the code. A column carries it — `last_seen_at` on the symbiot row, migration 0020, nullable and forward-only — and the whole store is two functions over it: `mark_seen` stamps `now()`, `is_active` reads it against the window the database compares with its own clock, so a null reads absent and a stamp older than the window reads absent too, without a sentinel instant pretending either way. The heartbeat is laid on the `/inbox` poll, in the request's own transaction, so it lands in lockstep with the read it rode in on — and only when the poll names a real session, so an anonymous `/inbox` marks no one and the poll can't be turned into a way to flip a symbiot present unauthed.

The filter itself went where the design said it would, as a third silent narrowing beside the two `dispatch` already had, but the shape that kept it honest was making it opt-in from the call site rather than a blanket rule. `dispatch` grew one keyword — `suppress_when_present` — and the presence read only fires when it's set, so the reminder path, which never sets it, pays nothing for a signal it doesn't use and can never have an explicit "email me at nine" silently eaten because the symbiot happened to be looking. The two unprompted fan-outs opt in — the missive `deliver` raises and the enrichment follow-up — and they're the only two, which is exactly the "unrequested courtesy versus named intent" line the design drew, now a one-word difference at each call site instead of a comment hoping to be honoured. When the switch is on and the symbiot is present, both out-of-app channels drop, because both *are* out-of-app — there is no in-app channel to keep, the in-app reach being the poll itself — so the nudge collapses cleanly to the durable record the poll is already surfacing. And the filters stack rather than compete: a channel globally disabled stays disabled, a channel held for presence stays held, and neither overrides the other.

Green, 308 up from 296. The twelve new tests pin the three things that had to be true and weren't obvious: the store reads absent on null and on a stale stamp and present only inside the window; the `/inbox` poll actually leaves the mark and an anonymous one doesn't; and — the crux — `deliver` to a present symbiot sends no email while the record still stands in `/inbox`, so the guarantee is untouched and only the redundant ping is spared. What the suite still can't reach is the same live seam everything else in this session owes: a real phone that has gone quiet, a real mailbox that should stay silent while the tab is open and should ring the moment it isn't. The machinery is built and proven at every layer the tests can hold; the last mile is by-hand, like the rest.

## letting a thought have more than one line: Enter breaks, a send control commits

The next grievance was one I'd been working around without naming: the prompt could only ever send a single line. Enter submitted, and that was the only newline I ever got — the one that ended the message and fired it. So a thought that wanted two lines, a list, a break for breath before the point, had nowhere to put the break; I either ran it all together or sent it in pieces. The input was single-line by construction, not by choice, and it had quietly decided for me that everything I say is one line long.

So Enter stops sending and starts making line breaks, and a send control takes over submission. I weighed keeping Enter-to-send with Shift+Enter for the break — the desktop chat convention — but it splits badly on a phone, where there's no Shift+Enter to reach for, and I live in the shell on my phone as much as my laptop. So the cleaner call was the one that reads the same on every device: Enter always breaks the line, the send control always commits, no gesture meaning two different things depending on what keyboard I happen to be holding. It costs the muscle memory of Enter-to-send for a quick command, and I took that cost on purpose for a rule that never surprises.

The seam was the same one the checklist and the landmark both taught me to respect: the terminal is hand-built, and its own doc draws the line that the terminal owns the *surface* and the *raw key handling* while the app owns what a line *means*. Submission is raw key handling, so the whole change lives in the terminal and the app above never learned that Enter changed its mind. The pretty part is that the editable is a `plaintext-only` contenteditable, which means the browser already knows how to insert a newline on Enter — I didn't have to insert one, I had to *stop swallowing* it. The old code caught Enter two ways, a `keydown` for the physical keyboard and a `beforeinput` for the soft one, and committed on each; I pulled both, and now Enter falls straight through to the browser's own editing on either path. `getLine` stopped stripping newlines out — they're content now, not noise — and `commit` sheds only a trailing run of them, so a stray Enter left dangling before I hit send doesn't tack a blank row onto what lands.

The send control is deliberately quiet furniture, not a chunky web button. It wears the terminal's own vocabulary — a dim `[ send ]` in the same bracketed, tick-on-hover style the checklist's `[ save ]` already speaks — pinned to the right edge of the input line, riding a lane the input's right padding holds open so a long line wraps before it rather than running underneath. And it only appears when there's something to send: a blank line — empty, or only whitespace and newlines — has nothing to commit, so the affordance stays hidden and surfaces the moment I type, gated by the same input-change hook that recomputes the ghost. The one thing I would not leave soft is the phone: a tap on the control swallows its own `mousedown` before it can steal focus from the editable, so the soft keyboard stays up across the send instead of collapsing and forcing me to tap back in for the next line. Pointer and keyboard reach the same commit; neither is the poor relation.

The kernel didn't move; this is all shell, and small — Enter unhandled, the two commit-on-Enter listeners gone, newlines preserved through `getLine` and trimmed at the ends in `commit`, a `send` span, and a few lines of CSS. And because it's a browser-and-input thing the suite can't reach, I proved it the way that surface has to be proven: I drove the built bundle in a headless browser, focused the live editable, and typed through the real input pipeline — insert a line, press Enter, insert another — then read the DOM back. Enter holds both lines together instead of sending, the editable keeps its content when Enter is pressed rather than clearing, the send control commits the whole multi-line block and clears the input, a trailing Enter is shed so the echo reads clean, the control shows only once there's something to send and hides again on a blank or whitespace-only line, and the control itself renders as pressable furniture — green, bracketed, dim at rest, pointer on hover, pinned to the right edge. Eighteen checks, all green against the real build, not a reasoned guess about what the DOM would do. The Joy is v0.0.5.

## the first cut into the deeper rot: letting time survive the fold

The urgent bug I named last session and refused to pretend the render tweaks had cured has three heads, and this is the first of them: the fold flattens time. When a conversation grows past the size the verbatim tail is kept to, its oldest turns are folded into the Gist — the one running paragraph that stands in for everything older than the tail — and every reply older than that boundary is read back through it. The trouble is what the fold hands the model. Every turn on the stream carries the instant it was said, and every other place that renders those turns already stamps them — the verbatim tail a reply sees reads `[Sun 21:10] The human symbiot: …`, the clock right there on the line. The fold is the one render that strips it: it builds the transcript as a bare `speaker: text` and then asks the model to write "continuous prose in the past tense." So the one input that most needs order is the one handed none, and distinct-time statements melt into a single timeless paragraph — and because each fold seeds the next, that flattening compounds every pass until the Gist cannot say "I told it one thing on Monday and the opposite on Tuesday" even in principle.

The fix is not a new machine; it's giving the fold the clock every other render already wears. Each turn in the fold's transcript gets stamped with the local time it was said, read in the symbiot's own zone the same way the reply path reads it. One difference from the tail: the tail shows only a weekday and a clock because it's recent and leans on the prompt's "now" line for the month and year, but the Gist spans arbitrary stretches — turns days or weeks apart fold in together — so a bare weekday there is ambiguous. The fold's stamp carries the full date, `[Mon 14 Jul 2026, 21:10]`, so a turn can never be pinned to the wrong week. And the merge prompt, which today says only "past-tense prose," learns to keep the order the stamps give it: hold events in the sequence they happened and keep the time-anchor on any fact whose time matters, so that when this summary is folded *again* it still reads as a timeline rather than a heap. The zone the stamp needs is already at the fold's call site — the background sweep has the connection and the symbiot in hand — so it reads the timezone with the same call the reply does and passes it down. Two files move: the fold itself, and the one line that calls it.

One line I'm holding, because it's the boundary between this head of the bug and the next. I'm keeping the Gist as prose and trusting the model to carry order through it, not restructuring it into a timeline of stamped records with a field per fact. The heavier, structural cut is real work, but it's the same cut that *supersession* needs — the second head, where a later fact has to override an earlier contradicting one — and I'd rather not build half of that machinery here under the wrong name. This pass earns exactly one thing: time survives the fold. If prose-with-anchors turns out too soft to hold ordering under repeated re-folding, that's the signal to go structural — but I'll ship the honest small cut and watch it before I build the big one.

## what two days of living in it actually cost: €42, and where it went

Two days. That's how long I lived inside it before the meter read €42, and that number is the one that stops a project — not a bug, a bill, the kind of running cost that quietly decides whether a thing gets to keep existing. So before touching anything I did the honest thing and mapped where I spend, grounded in my own code rather than a hunch, because you can't cut a cost you haven't located.

Here is what I found, and it was hiding in one line of config. Every generative call it makes runs on the same model — `glm-5.2` on Scaleway, its most capable and most expensive tier. Not because anyone decided each job needed it, but because the assignments cascade: the router's model is `glm-5.2`, the reply defaults to the router's, and the enrichment gate, the tool confirmation, the conversation fold, the tool decision all default in turn to one of those two. Six distinct jobs, one premium model, by inheritance rather than intent. And those six jobs are not the same kind of work at all. Composing a reply in its own voice, or folding a conversation's history into the one paragraph every later reply leans on — that is prose the symbiot lives inside, and it earns the good model. But deciding whether two concept names mean the same thing, or which of five tools a message is asking for, or scoring a handful of recalled candidates against a fact — those are bounded, schema-shaped, temperature-zero classifications, yes/no and pick-one judgments, and I have been paying flagship prices to make them. Worse, they are the *frequent* ones: the write path fires several router judgments per message as it files each fact, so the cheapest-to-serve calls are also the highest in volume. I was spending the most on exactly the work that needed the model least.

The turn of it, and the reason this is a good afternoon rather than a grim one, is that the machinery to route differently already exists — I built it earlier this session without yet using it for what it's best at. Models live in a durable catalog and roles map to models in a table an operator can rewrite through `/models`, so "which model plays this job" is already durable, per-role state rather than a hardcoded constant. What's missing isn't the mechanism. It's the models to route *to*, and the assignments that send each job to the right one.

The shape we settled on is three tiers of intelligence, not one model and not two. Two isn't enough, because it forces two genuinely different kinds of work into one bucket: composing prose I speak, and pulling a structured decision out of a message, are not the same job even though neither is a bare yes/no. So three rungs. A flagship rung for the voice and the load-bearing memory — the reply; the fold whose Gist every later reply inherits; and the enriched follow-up, because deciding whether a deeper reach has found something worth interrupting for, unprompted, is the fine judgment a weak model fumbles into either nagging or silence, and a bad unprompted interruption is worse than none. A middle rung for competent language and structured extraction — a tool confirmation spoken back in voice, a tool decision that reads five descriptions and picks one, and the minting of a new type's *definition* (generative, and wrong forever if it's wrong once, so it climbs out of the cheap bucket its sibling calls sit in; it fires only when nothing existing fits, so its volume is low and thins further as the vocabulary fills, and the middle rung's cost is easily worth the definitions everything downstream routes against). And a small, fast rung for the bounded temperature-zero classifications that dominate the write path by sheer volume — the yes/no and pick-one judgments the router fires several of per message: re-ranking recalled candidates, breaking the grey-zone tie, pulling the concepts out of a fact, reading its time, and, in the offline sweeps, judging two types the same kind and picking the survivor. Which actual models sit on each rung I'm leaving open on purpose: the catalogs and the per-token prices on both clouds are being checked before anything is named, because that check is what decides which cloud serves which rung, and a stale guess about pricing is exactly the wrong thing to build a cost cut on.

The flagship is the one rung that doesn't move: it stays `glm-5.2` on Scaleway, the capable tier the box already ran. The flagship work is the part where quality isn't negotiable — the reply the symbiot reads, the enriched follow-up that interrupts unprompted, and the fold whose Gist every later reply inherits — and `glm-5.2` already composes those well; there is no saving worth making by trading it down where the words matter most. Scaleway serves it at a 262,144-token window and a 16,384-token output cap, and I hold its input to the effective 131,072 the flagship-class models all sit at, because the advertised window is never the window a model reads *well*. So the flagship holds, and the whole cost pass is about the work *beneath* it.

Beneath the flagship, both cheaper rungs land on one model for now: `gpt-oss-120b`, OpenAI's open-weight mixture-of-experts, served on Scaleway at roughly a tenth of a flagship's per-token price. The reason it fits the middle and the small rung both is the thing those rungs most need — not eloquence but obedience. It sits at near-o4-mini parity with high-level reasoning and, the part that matters here, strict instruction-following, and every job below the flagship lives or dies on exactly that: the small rung's judgments are decoded straight into a Pydantic schema and must come back shaped exactly so, and the middle rung reads a set of descriptions and returns one structured decision. A model that follows the instruction and honours the format is worth more to those calls than a model with a finer voice, and this one follows well at a price that makes the write path's sheer volume affordable rather than alarming. Mid and small share it today rather than each getting their own model, but the three-rung *roles* stay split — so the door is open to drop the small rung onto an even cheaper model later (its smaller sibling, once that lands on the per-token endpoint rather than only dedicated inference) without disturbing the middle, and `mint` in particular stays up on the middle rung's model where a generative call belongs even while the two rungs happen to resolve to the same name.

The other cloud fills in beneath each Scaleway primary, and here the rungs Scaleway collapsed pull back apart. Mistral carries a genuinely graded lineup, so the fallback tiers itself the way the single Scaleway model couldn't: the middle rung falls to `mistral-small-latest`, the small rung to `ministral-8b-latest`, each a same-weight catch for the primary above it rather than one model standing in for both — a small classification that falls through summons a small model, not a middle one, so cost stays proportionate even when Scaleway is the cloud that's down. The windows keep the same half-the-advertised discipline as the rest: `mistral-small-latest` advertises ~256K and is held at 131072, `ministral-8b-latest` advertises 128K and is held at 65536, both on the reasoning that a model reads well across only about half the window it will accept. And both sit at or above the window of the Scaleway primary they catch — 65536 rising to 131072 for the middle, 65536 meeting 65536 for the small — so a prompt fitted for the primary can never overflow the model that inherits it when the ladder falls. Output stays capped at 16384 across both, the one ceiling the whole catalog already shares. The flagship's own catch stays the incumbent `mistral-large-latest` it has always fallen to — held, as it always has been, at 131072 and a 16384 output ceiling. With these three named, the ladder keeps a same-rung fallback on the other cloud beneath every rung, flagship and cheap alike, so provider-independence reaches the whole way down the tiers rather than living only at the top where it began.

One rule of the ladder is locked regardless of which other models land: the cloud rungs tier, but the local Ollama tier does not. When both clouds are unreachable, every tier — flagship, middle, and small alike — falls to a single local model, not three. On-box compute costs nothing, so there is no cost argument for mirroring the tiering down there; the local tier exists to keep it answering when the clouds are gone, not to save money, so one model is the whole safety net. The code already leans this way — the last resort is one fixed local name today, `qwen3.5:4b` — and the three-tier design keeps it that way rather than growing a local model per rung. So the fallback stays provider-independent at every cloud tier, with a same-rung model on the other cloud beneath each primary, and then collapses to one floor beneath all of them. Analysis first: every call site is mapped and the tiering is drawn; naming the models and locking each role to its rung comes next, because the cost of getting this wrong is either a bill that doesn't come down or a reply that quietly gets dumber, and I want neither by accident.

## the second head, and why I'm not cutting it here: supersession folds into identity

Back to the three-headed bug, and the second head: it keeps no notion of a later fact overriding an earlier one that contradicts it. "I haven't eaten tonight" and "I ate this afternoon" sit in the diary as coequal truths, and recall may surface the stale one as readily as the live one — so it can tell me I've eaten while I'm sitting there hungry. The obvious fix is a stale-or-live flag on a fact: the newer one wins, the older is marked past but kept. I turned it over and I'm not building it, because it's wrong twice over.

It's wrong first on the shape. A stale bit assumes a fact dies once and stays dead, and a life doesn't run that way. I lived in Strasbourg, then elsewhere, then Strasbourg again; a thing I quit I picked back up. A fact is true across a window, sometimes more than one window, and it can come back — none of which a single dead/alive bit can hold. It works for the short contradiction told across one evening and breaks over any endeavour that runs for months, which is exactly the span the diary exists to remember. Validity intervals would model it honestly, but that is more machine, not less, and I'm not reaching for more machine to fix a thing I haven't proven the cheap way can't hold.

It's wrong second, and more deeply, on the foundation. To say fact B supersedes fact A the machine first has to know they are about the same thing — the same address, the same project, the same person. That "same thing" judgment is the third head: the identity model. Built on today's raw-text-blob facts, with no entity underneath them, supersession is just guessing at which pairs collide, and it would guess wrong the moment two people or two projects sound alike. So supersession isn't a peer of the identity work; it sits on top of it. Once a fact hangs off a first-class entity, "the latest fact about this entity's address wins" is nearly free to express and needs no new machine at all. Trying to build it first is laying brick on sand — the same trap I stepped around on the first head when I kept the fold as prose rather than half-building the structural cut under the wrong name.

So the second head is dropped as a pass of its own. It isn't abandoned; it's re-parented. It rides on the identity model, and until that lands the cheap read-path anchoring already shipped — facts rendered in time order, the reply handed the date so it speaks "as of Tuesday" rather than flatly — carries the load it can. I'd rather do the one honest foundation than two half-foundations that each assume the other exists.

## and then done, ASAP because the meter was running: the cost cut goes in

The €42 was never a number to study for a week. The instruction that came with it was blunt and right — save money now — so this shipped as a deliberate first cut rather than the whole ladder polished at once: get the spend down today, and leave the refinement that only matters during an outage for its own pass.

What moved is small, because the machinery was already built. The role-to-model store and the `/models` command have been there since earlier this session; all that was missing was the models to route to and the assignments that send each job to the right one. So three tier constants now name the flagship, middle, and small rungs, and the roles resolve through them: the reply, the enrichment gate, and the conversation fold on the flagship, which stays `glm-5.2`; the tool confirmation and the tool decision onto the middle (`gpt-oss-120b`); the router's classifications onto the small rung (`gpt-oss-120b` too, for now). The cheaper models joined the catalog — `gpt-oss-120b`, and the two Mistral catches for the deferred fallback — while `glm-5.2` stays exactly where it was, the flagship the reply and the memory still lean on. And minting was cut out of the shared router default into a role of its own, so the one generative call in the write path rides the middle rung where a coined definition belongs, even while the middle and small rungs happen to resolve to the same model today.

The subtle part was making it land on the box already running, not just a fresh install. The seed only fills a role that has no assignment yet, on purpose — so an operator's choice is never trampled by a later boot — which means changing the defaults in code does nothing to a store already seeded with the old ones. So the cheap roles, which had all inherited the flagship model, were moved across to `gpt-oss-120b` on the running box directly, while the flagship roles stayed on `glm-5.2` where they already sat; a fresh box seeds the same split straight from the new defaults. I checked it landed rather than assuming it had: the live store now reads the reply, the enrichment, and the fold on the flagship and everything else on the cheap model, the kernel came back healthy on the reload, and the whole suite — three hundred and ten checks — stayed green.

One thing I held back on purpose, and it's worth naming so the next pass knows where to start. When Scaleway is down, every rung still falls to the same Mistral catch it always did — `mistral-large-latest` — rather than the cheaper same-rung model the plan calls for, `mistral-small-latest` for the middle and `ministral-8b-latest` for the small. The flagship's fall to `mistral-large-latest` is exactly right; the cheap rungs falling there is not, and it has teeth: an outage would send the high-volume classification calls stampeding onto the priciest fallback at the worst possible moment. The reason it isn't fixed here is that the fix needs the *rung* threaded through the generative boundary — the middle and small rungs share one primary model, so the ladder can't read the right fallback from the model name alone, the way it does today. That's a real, separable change, and the savings I came for land without it: the fallback only bites during an outage, and the steady-state spend — which is nearly all of it, nearly all the time — is already off the premium model. So this cut earns its keep now, and the rung-aware catch is the next one, named and waiting.

## the check that caught a live break: every rung, temperature zero and thinking off

New models mean old assumptions have to be re-proven, not carried over, so before calling the cost pass done I checked the thing that would quietly rot the replies if it slipped: that every rung is still driven the way this boundary has always promised — temperature pinned to zero, and thinking off. Temperature held up without a scratch: it's pinned to zero at the one place every call passes through, judgment and reply alike, and each provider adapter sets it from there, so there was never a rung where sampling drifted. Thinking is where the swap drew blood.

The old code disabled reasoning one way for everyone — Scaleway's `reasoning_effort="none"` — because the one Scaleway model it ever spoke to, GLM, accepted that. gpt-oss does not. Asked for `"none"` it returns a 400, "this model does not support none", and a 400 is our-side, not an outage, so it does not fall down the ladder — it just fails. Which means the moment I pointed the middle and small rungs at gpt-oss, every classification and every tool decision was failing outright: the reseed had quietly broken the whole write path, and I only know because I checked the live endpoint instead of trusting the code that used to be right. gpt-oss's reasoning floor is `"low"`, not off — but at `"low"`, for the bounded calls made here, it emits no reasoning trace and spends no reasoning tokens, so `"low"` is its effective thinking-off. So the effort is per model now: GLM takes `"none"` (confirmed live — a clean answer, no trace, no reasoning tokens), gpt-oss takes `"low"`, and the value sent is always one the model's own schema accepts, so a 400 on that field is impossible by construction. I proved the fix the way I found the break — a real structured-output call through the kernel's own code on the small rung, which now comes back clean.

The Mistral tier had its own buried assumption, and it was wrong too. I'd written that Mistral has no reasoning trace to suppress; that holds for `mistral-large-latest` (its reasoning variant hasn't shipped, and it rejects the effort field outright), but not for the cheaper rungs' catches. `mistral-small-latest` is Mistral Small 4, which introduced configurable reasoning, and `ministral-8b` ships in a reasoning variant as well. Right now this is latent, not live: the only Mistral model the ladder actually reaches today is `mistral-large-latest` — the cheap rungs still fall there, not to their same-rung Mistral catches — so every model the running ladder can touch is genuinely thinking-off. But it's a live wire for the rung-aware fallback that's still to come: when `mistral-small-latest` is wired in as the middle rung's catch, it has to be sent `reasoning_effort="none"` (it accepts it, verified) or it will think on the fallback path. That belongs with the rung-aware work, and it's written down there now rather than left to be rediscovered.

So the verdict is clean, with one honest asterisk: every rung the kernel can invoke today runs at temperature zero with no reasoning trace — flagship GLM on `"none"`, the cheap rungs' gpt-oss on `"low"`, the local floor on `think=False`, and the one reachable Mistral fallback non-reasoning by nature — and the whole suite, three hundred and ten checks, stays green. The asterisk is the not-yet-wired Mistral catch, whose reasoning-off is now part of the fallback pass rather than a surprise waiting in it.

## the shape of what moved: about three-quarters of the calls, by count

I wanted a rough sense of how much traffic the split actually redirects, so I counted the calls a single message sets off rather than guessing. The write path is the driver, and it fans out: filing one message reads its clock once, names the concepts it expresses (usually around three), and then routes each concept on its own — a re-rank apiece, plus the occasional grey-zone tie-break — so a typical message spends five or six small-rung calls before it is filed. The middle rung is far quieter: minting fires only for a genuinely new concept and thins toward nothing as the vocabulary fills, and the tool calls fire only when a message actually reaches for a tool, so the middle averages well under one call a message. Against all that the flagship spends about two: the reply, and the enrichment gate that judges whether to follow up. So of the roughly seven or eight generative calls a message now triggers, about three-quarters land on `gpt-oss-120b` — nearly all of them the small rung's router — and about two stay on `glm-5.2`. Before this pass every one of those calls was `glm-5.2`; now the premium model handles only the two that compose words the symbiot reads.

Two honest edges on that number. It is a count, not a bill: the two flagship calls carry the heavy prompts — the persona, the recalled diary, the conversation history folded in — while the five or six cheap calls are small and bounded, so the spend does not fall by three-quarters even though the call *count* does; the flagship still carries real weight for what it reads, not how often it runs. And the mix drifts: minting and the grey-zone gate both fade as the ontology stops meeting new concepts, so the per-message count settles toward the steady handful — read the clock, name the concepts, re-rank each — that files a fact into a vocabulary it already holds.

## keeping /models role-based: the rung is how it's built, not a knob I hand over

When the roles split into three tiers, the obvious next question was whether the `/models` command should let the operator pick a *rung* — set "the mid tier" to a model in one move — rather than assign each job on its own. I decided no, and it is worth saying why, because the temptation is real and the reasoning is the one that keeps the rest of the config honest.

The rung is not a thing the running box knows. Flagship, middle, and small are how the *defaults* are organized in code — three constants the role assignments are seeded from on first boot — and once a role has a row in the store it points straight at a model; the rung it came from is forgotten. So there is no durable tier to hand the operator a handle on. The operator's unit of control is the role — which model composes the reply, which model reads a fact's time — and that is *finer* than the rung and already covers everything selecting a rung could: to move the whole middle tier you assign its three roles, and a rung-level control would let you do nothing the per-role control can't.

Exposing rungs would cost one of two things, and neither earns its keep yet. Either a convenience macro — "assign the mid rung", fanning out to its roles — which needs the code to carry the rung-to-roles grouping as data it does not keep today; or first-class tiers, a role resolving through a tier to a model, two hops where there is now one, and a resolver rebuilt around a concept the operator would then have to learn. So the role table stays a flat, legible role-to-model map, the same shape it has always had, and the rung stays what it actually is: my own lens for deciding what deserves the capable model, not a dial bolted onto the panel. If reassigning the same group of roles together ever becomes a chore worth automating, the macro is the small answer to reach for then — built when the tedium is real, not in anticipation of it.

## a rule for how it looks: terminal-feel is the ground, not a cage

But the symbiot who lives here is pragmatic about the way a thing feels in the hand, and said the honest thing plainly: the terminal look is the ground I build from, not a wall I refuse to climb. Where lived experience makes a fuller surface the better one, straying from the register is allowed — the feel is a starting posture, not a vow.

So the emoji selector leans toward a full grid picker, more furniture than terminal purism would reach for, chosen because picking a face out of a grid is simply how the hand expects to do it and a shortcode you must know by name is not. The checklist already proved a modal surface can live in the log without betraying the feel, so this is sanctioned ground rather than new ground. Making the symbiot's own echoed lines read dimmer than the machine's replies, earlier this session, kept faith with the ground; the grid picker is the first deliberate step off it. ## the reminder that fired two hours early: a wall clock is not a UTC reading

I told it I had an hour and a half before the café closed, and it set a reminder and said it back: fires at 18:05. It was already past half six on my own clock. The hour it quoted sat two hours behind the one I was living in — and worse than the words, the reminder itself went into the store two hours early, so it never waited for closing at all; it fired the instant the machine made it, telling me the café shut in ten minutes while I still had ninety. A reminder that speaks the wrong hour is a nuisance; one that fires at the wrong moment is useless, and this managed both at once.

The pull was to blame the reminder machinery, and the machinery was innocent. I ran the whole decide-store-speak path against a Paris zone dozens of times and it came back right every time; the timezone command genuinely writes and reads back Europe/Paris; a missing timezone database would have crashed the write rather than misspoken it, and a reminder did fire, so that wasn't it either. What settled it wasn't in the code at all — it was the row still sitting in the deployed store. Read straight off the table, its fire time was 18:05 Paris, stamped fired thirty seconds after the machine made it. The database, not my reading of the code, told the truth: the wrong instant went in, and it went in already past. I'd been circling explanations that all fit the words; the row fit only one.

With the row in hand the mechanism was plain, and it was the first thing I'd suspected and then argued myself out of. The model that reads my sentence into a time had done the timezone arithmetic itself — taken 20:05 on my clock, converted it to its UTC reading of 18:05, and handed that back bare, no zone attached. And the executor's old rule was that a time with no zone is the local wall clock — so it read that bare 18:05 as 18:05 Paris, the UTC value mislabelled as local, two hours early. The reason my first sweep missed it: the models running today happen to hand the time back with its offset attached, correctly, on the clean prompts I was probing with — sixty runs, not one slip. The deployed call, on a longer and messier message, pulled a trigger the clean prompts never did. So I trusted the code over the box, and the box was right; the lesson is the old one, that the live row is evidence and my confidence is not.

The fix had a trap in it that only testing found. My first move was to stop inviting the conversion — show the model my time as a plain wall clock and tell it not to convert to UTC — and that alone made things worse in a new direction: some models answered with the right local numbers but labelled them UTC, and the old store, trusting the label, converted them forward — two hours late instead of early. The offset a model attaches is a guess, and it guesses wrong in both directions.

So the rule is no longer to trust the offset at all. The executor takes the clock face the model named — the hour and the minute — and stamps my zone onto it, discarding whatever offset rode along, because my zone is ground truth and the model's label is only a guess. That single change makes every shape land right: bare, correctly-offset, and mislabelled all resolve to the same 20:05 Paris. Beneath it sits a guard that would have caught the incident on its own — a reminder is for the future, so a time that resolves at or before now was read wrong, and it refuses it and asks me again rather than store something that fires the moment it's written. And the decision prompt now shows the current time as a bare wall clock with no offset to convert against, and asks for the same in return. Three bricks, and the load-bearing one is the store: it no longer believes the model about which zone a time is in.

I proved it the way I found it — fed the store the exact bare-UTC value off the deployed row and watched it refused now instead of written early, then ran the live decision across all three cloud models and saw every one land within a minute of the mark and safely ahead of now. That check is written down as a by-hand smoke that holds the whole invariant — every offset shape to the one instant, the incident refused, the live path near and future — so it can't quietly rot back. One honest note: this lives in the code now, not yet on the box I spoke to. The running kernel still carries the old rule until it's redeployed; the fix is real, and it reaches me when the deploy does.
